ACI Simulator is valuable, but only if you use it for the right job.

## What it teaches well

Use it for:

- APIC GUI navigation.
- Tenant and application policy modeling.
- Contracts, filters, and EPG relationships.
- Access policy object relationships.
- L3Out policy structure.
- Faults, health, inventory, audit logs, and object tree inspection.
- API and automation practice.

## What it does not replace

Do not rely on it for:

- Real endpoint traffic.
- Packet captures.
- Hardware forwarding behavior.
- Real BGP or OSPF adjacency troubleshooting.
- Real firewall insertion behavior.
- Real VXLAN EVPN interoperability.
- Performance, scale, or failure testing.

For those, use EVE-NG, physical gear, or a vendor-supported lab environment.

## Best next labs

Pair the simulator with these labs:

| Lab | Why it matters |
| --- | --- |
| Nexus VXLAN EVPN fabric | Learn the non-ACI data center fabric model |
| FTDv firewall insertion | Practice north-south and east-west security controls |
| IOS XR border routing | Learn service-provider style routing at the edge |
| Linux endpoints | Generate real traffic and test application behavior |
| Two data centers | Practice interconnect, mobility concepts, routing, and failure domains |
| GitOps network automation | Make the lab reproducible and reviewable |

## A practical learning sequence

1. Learn ACI objects in the simulator.
2. Build the same application segmentation concept in EVE-NG with plain routing, switching, and firewall policy.
3. Automate the ACI version with Terraform or Ansible.
4. Automate the EVE-NG/network version with Ansible, Nautobot, and Git.
5. Compare both models: controller-based intent vs box-by-box networking.

That comparison is the real lesson. ACI is not just another CLI syntax. It is a different operating model for data center networking.
