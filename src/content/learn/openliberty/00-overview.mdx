This is a self-paced, opinionated track on building a real Java application with **Open Liberty** — from the moment you have an empty VM through to running an enterprise service bus next to your app. By the end you will have provisioned a development host, written a Jakarta EE 10 / MicroProfile 6.1 service, packaged it into a container, set up a git workflow that survives across machines, and integrated through a separate WSO2 Micro Integrator runtime.

The track is opinionated. It teaches the choices a Liberty team would actually make today — kernel-vs-full image, web profile vs full Jakarta EE, rootless podman, single-VM development, ESB as a sidecar container. Where alternatives exist, I name them and say why we didn't pick them.

## Who this is for

Engineers who:

- Know Java — generics, annotations, lambdas, the basics of CDI or a DI container.
- Have used Maven (or Gradle) for a project. We won't dwell on Maven fundamentals.
- Are comfortable on a Unix shell — SSH, basic systemd, `ps`/`grep`/`curl`.
- Are *not* trying to become a JVM internals expert or a Liberty performance tuner. This is the surface-area version.

If you've never written a Java service that listens on a port, work through a Spring Boot or vanilla JAX-RS quickstart first. An hour; come back.

## What you'll build

A small REST service called `insurance-app`:

- **Jakarta EE 10 web profile + MicroProfile 6.1**, running on **Open Liberty**.
- Packaged as a **podman container** based on the official `open-liberty` image.
- Sits **behind a WSO2 Micro Integrator** that routes `/insurance/*` traffic to it.
- **Source tracked in git**, with the VM (not the laptop) as the canonical workspace.

The application logic itself stays deliberately thin — one `/api/ping` endpoint. The track's value is in everything around the endpoint: how you wire it, where you run it, how you change it. By the end you'll have the skeleton on which a real insurance application — claims, underwriting, fraud — could be built.

## How to use this track

Three patterns work:

- **Sequential.** Start at 01, walk to 06. Best for first-timers.
- **Reference.** Jump to the module you need from the sidebar. Best if you already have a Liberty app and want one piece.
- **Pair-programming.** Open the track on one monitor, your shell on another, and execute as you go.

## Prerequisites and setup

- A **Linux host** you can run KVM / libvirt on. Module 01 walks through the install.
- **SSH from your laptop** to that host. A jump host is fine — module 01 covers the ProxyJump pattern explicitly.
- A **GitHub account**. You'll push the project from inside the VM in module 05.
- **No Java or Open Liberty installed up front.** Module 02 installs everything on the VM.

## A note on what's *not* here

This track doesn't cover:

- **Open Liberty performance tuning.** Different discipline. Use the official Liberty performance docs.
- **EJB or the full Jakarta EE platform profile.** We use the web profile deliberately — module 04 explains why.
- **Cloud deployment.** Module 07 sketches the OpenShift path; building it out is a separate track.
- **Spring Boot.** Adjacent ecosystem. Mentioned where the comparison helps.

The goal: get you fluent in the **full developer loop** for a Liberty service. Not just the code — the host, the container, the network, the workflow.
