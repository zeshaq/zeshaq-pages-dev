This is the working plan for the BRAC Bank POC. **The POC is delivered entirely on OpenShift.** Every workload — including the ones our internal dc-lab currently runs on plain VMs (Jenkins, Nexus, SigNoz, Kafka, Redis, WSO2) — is redeployed OpenShift-native for this engagement. No docker-runtime, no docker-compose, no plain-VM services in the POC plane.

## Why OpenShift-only

- The customer's evaluation explicitly anchors on OpenShift (Section 1 of the [requirements email](./requirements-email)). Showing the rest of the stack OpenShift-native makes the POC cohere as one platform story rather than a mosaic of provisioning paths.
- It removes a class of objection ("why is Kafka on a VM if you're an OpenShift shop?") and lets us demonstrate one set of operating patterns — Operators, CRs, namespaces, NetworkPolicies, Vault/ESO, GitOps — across every product.
- It limits the failure surface during evaluation. One control plane to demo, one log pipeline to inspect, one RBAC story to defend.

## Target environment

- **1 OpenShift cluster, 3 nodes, on VMs**, provisioned with Terraform per Section 1.Infrastructure. Sized to host every workstream simultaneously — masters consolidated with worker role for the POC (acceptable for evaluation; production sizing addressed separately).
- **ODF** (OpenShift Data Foundation) for Block + Object storage classes.
- **Internal services namespace strategy** — one tenant namespace per workstream so RHACS, NetworkPolicies, and resource quotas can be demonstrated cleanly:
  - `brac-observability` (OTel + SigNoz + ClickHouse)
  - `brac-identity` (WSO2 APIM CP/DP + WSO2 IS)
  - `brac-cicd` (GitLab + Jenkins + Nexus + ArgoCD instance scoped to POC)
  - `brac-trivy-dashboard` (DefectDojo on OCP — the one-dashboard view)
  - `brac-redis` (Redis HA)
  - `brac-streaming` (Kafka KRaft + Schema Registry + Connect)
  - `brac-middleware` (Open Liberty + NGINX)
  - `brac-jboss` (JBoss EAP managed domain)
- **Platform plane** (RHACS, Compliance Operator, Loki, Tempo, cert-manager, ESO + Vault, Argo CD) shared across all tenant namespaces.
- **Identity** — htpasswd + cluster-admin for evaluators; WSO2 IS for application-layer federation demos.

## Per-section delivery

### 1. OpenShift Platform foundation

- **Terraform module** at `infra/terraform/brac-poc-cluster/` that provisions a 3-node OCP cluster on VMs (libvirt or vSphere depending on what BRAC provides). Inputs: cluster name, base domain, machine network CIDR, pull-secret reference. Outputs: kubeconfig + cluster API + console URL.
- **ODF**: install via OperatorHub, create `StorageCluster` CR for Block + Object classes. Demonstrate by binding a PVC (block) and an `ObjectBucketClaim` (object).
- **Compliance Operator + PCI-DSS**: install operator, apply `ScanSetting` + `ScanSettingBinding` for `ocp4-pci-dss-4-0` + `ocp4-pci-dss-node-4-0`. Run scan; export the auto-remediation report (`ComplianceCheckResult` + `ComplianceRemediation` CRs) as a PDF artifact.
- **RHACS critical-block policy**: enable Central + Sensor, create a policy in the "Build" + "Deploy" lifecycles with severity `CRITICAL_SEVERITY` and `enforcementActions: [SCALE_TO_ZERO_ENFORCEMENT, FAIL_BUILD_ENFORCEMENT]`. Demonstrate by deploying an intentionally-vulnerable image and showing the scale-down.
- **Prom/Grafana custom alerts**: PrometheusRule CRs scoped per workstream namespace. At least three alert classes: workload health (pod restart loops), resource (CPU/mem usage > 80%), and security (RHACS violation count).
- **Logging pipeline**: LokiStack via the Loki Operator; ClusterLogForwarder forwarding `application` + `infrastructure` + `audit` log types to (a) in-cluster Loki and (b) an external mock destination (a TCP receiver pod in `brac-observability` to prove the forwarder works).

**Done = a single command (`terraform apply` + `argocd app sync`) brings the cluster up with all baseline operators installed and the PCI-DSS scan, RHACS policy, and logging pipeline running.**

### 2. Logging & Observability (OpenTelemetry stack)

- **Instrumented sample microservice**: deploy two pods of the same app — one with OTel SDK auto-instrumentation (.NET preferred per the dashboard requirement), one without SDK (relying on sidecar interception). Demonstrate both flows produce traces/metrics/logs in the pipeline.
- **Pipeline**: OTel Collector (Gateway mode) → Kafka topic per signal (`otel.traces`, `otel.metrics`, `otel.logs`) → SigNoz consumer → ClickHouse storage. Each signal has its own topic so consumers can scale independently.
- **Traffic management**:
  - **Trace sampling** at the Collector via the `tail_sampling` processor (e.g., 100% of error spans, 10% of OK spans).
  - **Log filtering** at the Collector via the `filter` processor (drop debug-level by default; configurable).
  - **Metrics segregation** at the Collector via the `routing` processor — workload metrics vs runtime metrics go to separate ClickHouse tables.
- **ClickHouse lifecycle**:
  - 2-day hot retention via `TTL` clauses on the relevant tables.
  - Cold archive: `s3` table function targeting an MinIO bucket (`brac-poc-otel-archive`), driven by a daily `clickhouse-backup` schedule.
- **Dashboards** (in SigNoz, exported as JSON committed to git):
  1. **Application Performance** — HTTP request duration p50/p95/p99, request count, active requests, DB connection pool usage.
  2. **.NET Runtime** — GC heap size, GC collection counts per generation, thread pool active threads.
  3. **System** — process CPU + memory per pod.
  4. **Tracing** — service dependency graph (SigNoz's native), top latency endpoints, throughput per service, span timeline view.

### 3. Identity & API Management (WSO2)

- **Distributed APIM**: WSO2 Helm chart with `controlPlane.enabled: true` + `dataPlane.enabled: true` deployed as two StatefulSets in `brac-identity`. CP fronts the Publisher + DevPortal; DP fronts the Gateway runtime.
- **External databases**: CloudNativePG (CNPG) `Cluster` CRs for the four WSO2 databases (`WSO2_SHARED_DB`, `WSO2_USER_STORE`, `WSO2_APIM_DB`, `WSO2_REG_DB`). 3-replica clusters with `instances: 3` for HA. Backup to MinIO via the CNPG `barmanObjectStore`.
- **Key Manager**: WSO2 IS deployed as a separate Helm release in `brac-identity`. Configured as the Identity Key Manager from the APIM Publisher.
- **SSO**: configure two demo Service Providers in IS — one SAML2 (for a sample SP webapp), one OIDC (for the APIM Publisher UI itself). Federate against an upstream demo IdP (a second IS instance acting as "Customer IdP") to prove identity federation.
- **Hardening**: drop default H2 (verified by exec into pods, no `repository/database/WSO2*_DB.mv.db` files), enable mutual TLS between CP and DP, configure the WSO2 logger to forward to Loki via the ClusterLogForwarder, add a RHACS policy to flag any pod still mounting `WSO2_H2`.

### 4. CI/CD & DevOps Tooling

- **GitLab on OCP**: GitLab Cloud-Native Helm chart with `psql.external: true` pointing at a CNPG cluster, MinIO for object storage, and Redis for session/cache. HA mode = `replicas: 2` on the web/sidekiq/registry components. Backup via the chart's built-in `gitlab-backup` cron + replication to a DR namespace.
- **Jenkins HA on OCP**: Jenkins Operator-managed Jenkins controller with `replicas: 2` (active-standby via leader election + shared PVC on ODF). Casc config from a ConfigMap, plugins pinned via a `plugins.txt` ConfigMap (today's `templates/jenkinsfiles/lib/defectdojo.groovy` pattern from #64-#66 applies unchanged).
- **Monorepo conditional Jenkinsfiles**: demonstrate via a sample monorepo with two app dirs (`apps/payment-api/`, `apps/risk-engine/`). Jenkinsfile uses `changedFilesPattern` + matrix to trigger only the affected app's build + scan. Reuses the `update-overlay-digest.sh` script that today's #234 work aligned to `dev|stg|prd`.
- **Hybrid CD**: Jenkins pipeline that deploys one path to OpenShift (Liberty app via OpenLibertyApplication CR) and one path to a legacy VM (mock VM target — a single docker-less Liberty instance reached via SSH + `systemctl`). Acceptance: same build artifact lands in both planes via the same pipeline.
- **ArgoCD**: dedicated ArgoCD instance in `brac-cicd` (separate from the platform ArgoCD running OCP infra). AppProject scoped to `brac-*` namespaces, ApplicationSet generating one App per workstream namespace from the GitOps repo, sync waves following the dc-lab convention (10 = operators, 20 = operands, 30 = wiring).
- **Sonatype Nexus on OCP**: Sonatype Helm chart with `database.external.host` pointing at a CNPG cluster, ODF block-storage PVCs for blob stores. Three repositories per format (Maven, PyPI, npm, Docker) — proxy + hosted + group — mirroring the three-endpoint split from dc-lab ADR 0019. HA via 2 read-write replicas (caveat: Nexus OSS doesn't do true active-active; document the limitation and demonstrate failover with the standby).

### 5. Trivy single dashboard (SCA + SBOM)

- **DefectDojo on OCP**: deploy the official DefectDojo Helm chart (not the docker-compose form) in `brac-trivy-dashboard`. External Postgres via CNPG.
- **Jenkins → Trivy → DefectDojo import**: reuse today's shipped pipeline (commit `b84037c`, `templates/jenkinsfiles/trivy-defectdojo-import.groovy` + `lib/defectdojo.groovy`). Configure Vault path `secret/platform/defectdojo/api-token` via ESO and a Jenkins credential.
- **SBOM section**: extend the existing pipeline to call Trivy with `--format cyclonedx` for SBOM generation; import the CycloneDX SBOM into DefectDojo as an `Anchore Grype` or `Generic Findings Import` and demonstrate the SBOM view in the DefectDojo UI alongside the SCA findings.
- **Dashboard view**: a single DefectDojo product per app with two engagements per build — "SCA" + "SBOM". Findings views configured so the evaluator sees both side-by-side.

### 6. Redis HA on OpenShift

- **Operator**: OT Containers' `redis-operator` (CRD: `RedisSentinel` + `RedisReplication`) or Redis Enterprise Operator (if a license is acceptable). Sentinel topology with `RedisSentinel.spec.size: 3` + `RedisReplication.spec.size: 3` (1 master + 2 replicas).
- **Networking**: ClusterIP Service for the Sentinel pool, separate Service for the data-plane (Redis primary), NetworkPolicy allowing only `brac-cicd` + `brac-identity` ingress.
- **Failover demo**:
  1. Deploy a mock client pod (Python `redis-py` with `Sentinel` constructor pointing at the Sentinel Service).
  2. `oc delete pod redis-replication-0 --grace-period=0 --force` (hard kill).
  3. Tail the Sentinel logs showing the new primary election.
  4. Tail the mock client's logs showing it reconnecting through Sentinel (typically &lt;30 seconds).
- **Evidence**: a recorded terminal session with the kill timestamp, the Sentinel `+switch-master` event, and the client reconnect.

### 7. Event Streaming & Governance (Apache Kafka)

- **Strimzi Kafka KRaft**: `Kafka` CR in `brac-streaming` with `spec.kafka.kraftMetadata: migration` or pure KRaft (no ZooKeeper deps). 3 KRaft controller nodes + 3 broker nodes (can be the same StatefulSet via combined roles for the POC). Storage via ODF `csi.ceph.com` block class.
- **Schema Registry**: Apicurio Registry (CRD: `ApicurioRegistry`) backed by Kafka topics for schema storage. Configured as the schema source for two clients: a producer pod and a consumer pod.
- **Schema validation demo**:
  1. Define an Avro schema for a `Payment` event.
  2. Producer publishes a valid `Payment` — accepted.
  3. Producer publishes a malformed `Payment` (missing required field) — Registry rejects via the `AvroConverter` schema-id check; the produce call surfaces the error.
- **Kafka Connect + DLQ**: `KafkaConnect` CR with two `KafkaConnector` CRs:
  - **Source**: `FileStreamSourceConnector` reading from a ConfigMap-backed file.
  - **Sink**: `FileStreamSinkConnector` writing to a PVC.
  - **DLQ**: `errors.tolerance: all` + `errors.deadletterqueue.topic.name: brac-dlq`. Inject a poison record into the source; tail the `brac-dlq` topic to show the failed message.

### 8. Middleware & Web Tier (Open Liberty + NGINX)

- **Two Liberty instances**: two `OpenLibertyApplication` CRs in `brac-middleware`. Same image, different `replicas: 1` each, different `app.openliberty.io/instance` labels (`a` vs `b`).
- **NGINX reverse proxy**: deploy NGINX as a `Deployment` (not the Ingress Controller — this is application-layer reverse proxy per the email). Config via ConfigMap with:
  ```nginx
  map $http_version $upstream_pool {
      "beta"  liberty-b;
      default liberty-a;
  }
  upstream liberty-a { server liberty-a-svc:9080; }
  upstream liberty-b { server liberty-b-svc:9080; }
  server {
      listen 8080;
      location / { proxy_pass http://$upstream_pool; }
  }
  ```
  Tested with `curl -H "version: beta" http://nginx-svc/...` → routes to `liberty-b`; everything else → `liberty-a`.
- **MicroProfile observability**: enable `microprofile-health-4.0` + `microprofile-metrics-4.0` features in the Liberty `server.xml` (via the CR's `applicationImage` build or a `ConfigMap` mount). Expose `/health` and `/metrics` through NGINX. Scrape `/metrics` from Prometheus; build a SigNoz/Grafana dashboard showing `vendor:thread_count` (JVM thread count) and the per-readiness/liveness probe status.
- **Hardening**: NGINX config adds:
  - `add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;` (HSTS)
  - `error_page 404 /custom-404.html;` + a custom 404 ConfigMap-mounted page.
  - `limit_req_zone $binary_remote_addr zone=loginzone:10m rate=5r/s;` on the application's login route to demonstrate brute-force resistance.

### 9. Enterprise App Platform (JBoss EAP managed domain)

- **EAP Operator on OCP**: install via OperatorHub. The operator provides the `WildFlyServer` CR; managed domain is exposed via the `domain.xml`-driven init path.
- **Topology**:
  - **Domain Controller**: one `StatefulSet` replica running `domain.sh --host-config=host-master.xml`.
  - **Host Controller**: one `StatefulSet` replica running `domain.sh --host-config=host-slave.xml` pointing at the Domain Controller's discovery Service.
  - **Server Groups**: two declared in `domain.xml` (`brac-eap-app-a` and `brac-eap-app-b`); each maps to a Deployment that the Host Controller spawns as a server instance.
- **Operating demo**:
  1. Show domain-mode CLI: `bin/jboss-cli.sh --connect`, `:read-children-names(child-type=server-group)` → returns both groups.
  2. Deploy a sample WAR to Server Group A only; verify B is unaffected.
  3. Roll Server Group A (`/server-group=brac-eap-app-a:reload-servers`); B keeps serving.
- **Caveat**: managed-domain mode is unusual in containers — typical OpenShift JBoss deployments use standalone mode. The plan explicitly demonstrates managed domain because the requirements call for it; we'll note in the deliverable that standalone is the typical production pattern for OCP and managed domain is shown for parity with the requirement.

## Phase sequencing

The 9 sections compose into 6 phases. Phases run sequentially; sections within a phase run in parallel.

| Phase | Wall-clock target | Sections | What lands |
|---|---|---|---|
| **P1: Foundation** | Week 1 | 1 | OCP cluster + ODF + RHACS + Compliance Operator + LokiStack + Prom alerts + log forwarder |
| **P2: Streaming + Storage** | Week 2 | 6, 7 | Redis HA on OCP, Kafka KRaft + Schema Registry + Connect on OCP |
| **P3: Observability stack** | Week 2-3 | 2 | SigNoz on OCP, ClickHouse with hot/cold, OTel collector pipeline, 4 dashboards |
| **P4: Identity + API** | Week 3-4 | 3 | Distributed WSO2 APIM + IS, CNPG backends, SSO flows |
| **P5: CI/CD on OCP** | Week 3-4 | 4, 5 | GitLab + Jenkins + Nexus on OCP, ArgoCD scoped instance, DefectDojo dashboard, Trivy SCA + SBOM imports |
| **P6: App Platform** | Week 4-5 | 8, 9 | Open Liberty × 2 + NGINX canary, JBoss EAP managed domain |

After P6: one week reserved for the **demo flow + evidence pack** (see below).

## Demo / acceptance flow

The evaluator should be able to follow this sequence end-to-end without any side conversations. Each step has a 5-minute screencast and a written artifact in the evidence pack.

1. **Cluster provision** — `terraform apply` from zero → working 3-node OCP cluster + console URL + admin login.
2. **Operator install via GitOps** — `argocd app sync brac-poc-platform` → ODF + RHACS + Compliance + Loki + Tempo + Pipelines + CNPG + Strimzi + Apicurio + EAP + Liberty all install via Subscriptions in sync-wave 10.
3. **Compliance scan** — kick `ScanSettingBinding`; export the PCI-DSS remediation report.
4. **RHACS critical block** — try to deploy a CVE-rich image; show the deployment scaled to zero and the policy hit recorded.
5. **Logging** — query Loki for an `audit` log from a `cluster-admin` action.
6. **Kafka + Schema Registry** — produce valid + invalid payments; show validation reject.
7. **Kafka Connect DLQ** — inject poison record; show DLQ.
8. **Redis hard-kill** — `oc delete pod` the primary; show the mock client reconnecting.
9. **SigNoz dashboards** — open all 4 dashboards, hit the sample microservice with load, watch metrics update.
10. **ClickHouse retention** — query the hot tier; query the archive bucket via `s3` table function.
11. **WSO2 SSO** — SAML demo + OIDC demo + Federation demo.
12. **GitLab → Jenkins monorepo conditional** — push a change to `apps/payment-api/`, watch only the payment build trigger.
13. **Hybrid CD** — same build deploys to OCP and to the mock legacy VM.
14. **Trivy + DefectDojo** — push an image with known criticals; show DefectDojo dashboard with both SCA and SBOM views.
15. **Liberty + NGINX canary** — `curl` with and without the `version: beta` header; show different upstreams.
16. **Liberty health/metrics** — open the dashboard, show JVM thread count + liveness toggling.
17. **NGINX hardening** — show HSTS header, custom 404, rate-limit hit on the login route.
18. **JBoss managed domain** — CLI domain navigation, server-group deploy, rolling reload.

## Evidence pack

Per section, a folder in `reports/brac-poc/<section-number>-<slug>/` with:

- A 1-page summary of what was demonstrated.
- The CRs and ConfigMaps applied (committed to `platform-gitops` under `clusters/brac-poc/`).
- Screenshots of the relevant UIs (RHACS, SigNoz, DefectDojo, WSO2 Publisher, JBoss CLI).
- Logs from the critical demo moments (Sentinel `+switch-master`, RHACS policy hit, DLQ message, Schema Registry reject).
- Recorded terminal sessions (asciinema or `script` output) for the live demos.

Nothing in the evidence pack contains tokens, kubeconfigs, or credential bytes — Vault paths are referenced by name only.

## What's intentionally not in this plan

- The dc-lab platform (`hub-dc-v6` + `spoke-dc-v6`) keeps running independently. The POC cluster is new; nothing in dc-lab is reused as the live system.
- No docker-runtime VM, no docker-compose deployments anywhere. Tools the dc-lab runs on docker-compose (SigNoz, DefectDojo) get redeployed via Helm charts on OCP.
- No oc-mirror for the POC. Image supply uses the same Nexus three-endpoint split deployed inside the POC namespace, or BRAC's existing registry if they prefer.
- No "Path A vs Path B" distinction. Jenkins is the single CI path for the POC. Tekton patterns from dc-lab may inform some Jenkins pipeline shapes but Tekton is not deployed.
- Sections 10–13 (if they exist in the un-truncated PDF) are out of scope until we see them.
