This page consolidates the two BRAC source documents into one plan: the requirements, the architecture, what we reuse vs build, the per-item delivery, the phase sequencing, and the live demo flow.

**Operating principle:**

- **Demo apps run on OpenShift** (spoke-dc-v6). Not on the `docker-runtime-vm` we built earlier for the broader app-dev path.
- **dc-lab platform VMs are reused as-is.** SigNoz, DefectDojo, Kafka KRaft, Redis Sentinel, WSO2 APIM/IS, Jenkins, GitLab, Nexus, Trivy — no Helm/operator rebuild of services that already run on lab VMs.
- **New work is OCP-side**: the demo apps + Service Mesh + OTel collector + integration glue (ESO bindings, Routes, NetworkPolicies, mesh canary, the few new operators).

The two source documents (preserved verbatim for the audit trail):

- [Requirements email](./requirements-email) — the 9-section tender qualifying scope.
- v2 Demo-Scope PDF (received 2026-05-11) — 8-item live-demo subset, adds Service Mesh + mesh canary, includes the MFE/BFF reference architecture image.

## Architecture (consolidated)

The v2 PDF's reference image translated to OCP + dc-lab terms:

```
User
  │ HTTPS (TLS terminate #1 — OCP Route + cert-manager)
  ▼
MFE Router               ───┐
  │ HTTP                    │ in `brac-demo-frontend`
  ▼                         │ namespace on spoke-dc-v6
WSO2 API GW              ───┘
  │ HTTPS (TLS terminate #2 — existing WSO2 VM, lab CA)
  ▼
BFF Router               ───┐
  │ HTTP                    │ in `brac-demo-frontend`
  ▼                         │ (mesh sidecar attached)
Mesh boundary ─────┐        │
  │ mTLS strict    │        │
  ▼                │        │
Identity ◀────── WSO2 IAM (lateral, token introspection / authz callout — existing WSO2 IS VM)
  │ mTLS           │
  ▼                │
Product            │
  │                │
  └─[mesh observability sidecars → OTel Collector → existing Kafka VMs → existing SigNoz VM + ClickHouse]
```

Pieces and their hosts:

| Layer | Component | Host |
|---|---|---|
| Edge TLS | OCP Route + cert-manager + lab CA | `spoke-dc-v6` |
| MFE Router | NGINX `Deployment` | `brac-demo-frontend` ns on spoke-dc-v6 |
| API Gateway | WSO2 APIM (existing distributed CP/DP setup) | dc-lab WSO2 VMs |
| BFF Router | Liberty or Node `Deployment` | `brac-demo-frontend` ns |
| Mesh data plane | OSSM 3 ambient | `brac-demo-services` ns (label `istio.io/dataplane-mode=ambient`) |
| Identity service | Open Liberty `OpenLibertyApplication` v1+v2 (canary'd) | `brac-demo-services` ns |
| Product service | Open Liberty `OpenLibertyApplication` | `brac-demo-services` ns |
| IAM (authz callouts) | WSO2 IS (existing) | dc-lab WSO2 IS VM |
| Trace + log pipeline | OTel Collector (Gateway) → Kafka → SigNoz | Collector on `brac-demo-otel` ns; Kafka + SigNoz on existing dc-lab VMs |

## Phase sequencing

Block A items are reuse-walkthroughs (no new code; just narration + screencasts). Block B is the actual build work, sequenced so highest-priority demo items land first.

| Phase | Wall-clock | What lands | Demo'able by end |
|---|---|---|---|

| **P1: Observability + Kafka pipeline** *(HIGH PRIORITY)* | Day 3-5 | OTel Collector deployed; sample microservice w/+w/o SDK; pipeline OTel→Kafka→SigNoz wired; 4 dashboards authored | End-to-end trace appears in SigNoz; sampling/filter/routing visible |
| **P2: Service Mesh + mTLS + canary** *(v2 priority)* | Day 5-7 | OSSM scoped to `brac-demo-services`; STRICT mTLS; VirtualService canary on identity-svc v1/v2; Istio→OTel trace export | Kiali shows mTLS green; canary curl demo passes |
| **P3: MFE + BFF + WSO2 wiring** | Day 7-10 | MFE Router (NGINX edge), BFF Router, egress NetworkPolicy to WSO2 VMs, SAML + OIDC + Federation flows verified end-to-end | Full user → MFE → API GW → BFF → mesh → Identity → Product chain |
| **P4: Redis + Trivy + CI/CD glue** | Day 10-12 | Redis HA on OCP (if Q1 = yes); SBOM pipeline extension; BRAC monorepo + conditional Jenkinsfile; hybrid CD to OCP + one VM target | Hard-kill failover; DefectDojo SBOM view; monorepo-aware build triggers only the affected app |

## Live demo flow (v2 priority order)

The live demo runs in v2 PDF order so the HIGH-PRIORITY items land first if time runs short. The tender items follow as a secondary block.

**Block 1 — v2 PDF priority subset:**

1. **API Gateway** (existing WSO2 APIM): publish a sample API, show the policy + key flow + IS-as-Key-Manager link.
2. **IAM** (existing WSO2 IS): SAML SP demo + OIDC client demo + Federation against a second IdP source.
3. **Service Mesh — mTLS** (new OCP): `istioctl authn tls-check` returns MUTUAL; Kiali shows all-green mTLS edges.
4. **Canary via mesh** (new OCP): `curl` with and without `x-canary: beta`; show traffic split + Kiali request graph.
5. **Observability + Kafka — high priority** (new collector + existing Kafka + SigNoz): hit the demo app, watch traces traverse identity → product in SigNoz, show Kafka topic lag, expand spans.
6. **Redis** (decision-dependent): if Q1=yes, demo OCP-side hard-kill failover; otherwise narrate existing VM Sentinel topology with mock client reconnect.
7. **SigNoz** (existing VM): tour the 4 dashboards (app perf, .NET runtime, system, tracing).
8. **Nexus OSS** (existing VM): show three-endpoint split; demo multi-format pull (Maven + npm + Docker) from the same Nexus.

**Block 2 — tender-only items (if time permits):**

9. **OpenShift Platform foundation**: walkthrough `scripts/rebuild/` profile-aware automation; ODF Block + Object PVCs.
10. **Compliance + PCI-DSS**: export latest ComplianceCheckResult + remediation report.
11. **RHACS critical-block**: try to deploy a CVE-rich image; show scale-to-zero.
12. **Prom alerts + Loki**: trigger a BRAC-scoped alert; query Loki for an `audit` log; show external-mock log forward.
13. **Trivy SCA + SBOM dashboard**: DefectDojo product view with both engagements side-by-side.
14. **GitLab + Jenkins monorepo conditional**: push to `apps/brac-payment-api/`; show only that build triggers.
15. **Hybrid CD**: same build artifact deploys to OCP (Liberty CR) and to one existing VM target.
16. **Kafka Schema Registry + DLQ**: produce a malformed Avro message; show Registry reject. Inject a poison record into Connect; show DLQ.
17. **JBoss EAP managed domain**: CLI navigation, WAR deploy to one Server Group, rolling reload.

## Evidence pack

Per workstream, a folder in `reports/brac-poc/<NN>-<slug>/`:

- 1-page summary of what was demonstrated + which dc-lab assets it consumed.
- CRs + ConfigMaps applied to `spoke-dc-v6` (committed to `platform-gitops` under `clusters/spoke-dc-v6/brac-demo/`).
- Screenshots of the relevant UIs (RHACS, SigNoz dashboards, WSO2 Publisher, DefectDojo, Kiali mesh graph, JBoss CLI).
- Recorded terminal sessions (asciinema or `script`) for the live demos.
- Logs from the critical moments (Sentinel `+switch-master`, RHACS policy hit, canary traffic shift, mTLS handshake, Schema Registry reject).

No secret values, no kubeconfigs, no credential bytes in the evidence pack. Vault paths referenced by name only.

## Open questions

1. **Redis on OCP vs reuse VM Sentinel:** BRAC says "deploy on OpenShift Platform." Default in this plan: deploy a small Redis HA on `spoke-dc-v6` *in addition to* the existing VM Sentinel. Confirm or override.
2. **OCP cluster choice:** use existing `spoke-dc-v6` (default — fastest) or stand up a separate POC cluster (more demonstrable for the Terraform-automation requirement but adds 2-3 days)?
3. **OCP version pin:** confirm BRAC's evaluation tools work against the current spoke-dc-v6 minor; if they want a specific version, plan a controlled upgrade or a side cluster.
4. **JBoss EAP image entitlement:** valid pull secret for the EAP Operator's images? Otherwise fall back to community WildFly.
5. **WSO2 license / image entitlement:** existing WSO2 VMs run fine. Confirm no new license needed for the POC.
6. **DC-DR scope:** default is "documented procedure + one rehearsed restore in a side namespace" rather than a second live cluster. Confirm.
7. **Truncated requirements PDF:** original email mentions "13 Products" but PDF ends at section 9. Request sections 10-13 before P5 finalizes.
8. **MFE / BFF stack choice:** NGINX-only for MFE, or a real micro-frontend aggregator (Module Federation / single-spa)? BFF as Liberty (Java ecosystem fit) or Node (lighter)?
9. **"Comply MTLS" interpretation:** v2 PDF says "MTLS (Commply MTLS / Service mesh)". Likely typo for "Compliance-grade mTLS" — strict mTLS + FIPS-ish posture + audit. Confirm.

## What's intentionally not in this plan

- **No rebuild of dc-lab services on OCP.** SigNoz, DefectDojo, Kafka KRaft, Redis Sentinel, WSO2 APIM/IS, Jenkins, GitLab, Nexus, Trivy — all reused on their existing VMs.
- **No deployment of demo apps to `docker-runtime-vm`.** That path exists for the broader app-dev direction (`project_app_dev_direction.md`), but for this POC the demo apps run on OCP.
- **No second OCP cluster** unless Q2 = yes.
- **No NGINX-based header canary** at the application tier — replaced by mesh canary per v2 PDF (the MFE Router still has NGINX hardening: HSTS + custom 404 + rate-limit on login).
- **No new ADRs** unless a load-bearing direction changes (e.g., adopting OCP-native Redis or migrating Kafka off VMs as strategic direction beyond the demo).
- **Sections 10-13** of the original requirements PDF — not in scope until we see them.
