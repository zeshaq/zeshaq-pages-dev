## What it is

Four apps, two namespaces of concerns, one tenant namespace
(`bank-payment`) on **spoke-dc-v6**:

| App | Tech | Purpose | Users |
| --- | --- | --- | --- |
| `payment-client-svc` | Open Liberty / JAX-RS / MP-JWT 2.1 | Customer-facing payment API. System of record for accounts + transfers. | `zahid`, `shaikat` |
| `payment-approver-svc` | Open Liberty / MP REST Client | Admin-only API. Pending queue, approve / decline, audit. Calls the client svc's internal endpoints with the admin's own token. | `admin` |
| `client-frontend` | React 18 + Vite + Tailwind v4 + `react-oidc-context` | Customer SPA. Balance, send-payment form, transfer history. | `zahid`, `shaikat` |
| `approver-frontend` | React 18 SPA | Admin SPA. Pending queue with inline Approve / Decline; audit log. | `admin` |

Realm: `bank-payment` on `wso2-is.sub.comptech-lab.com`. Three users
(`zahid` / `shaikat` / `admin`) with one shared password
(`Hkj38djf&&&`) — `zahid` and `shaikat` carry an `accountNumber` plus a
seed balance; `admin` does not own an account, only the approver role.

**Reading the diagram:**

- **WSO2 IS** issues a JWT to either SPA via OIDC auth-code + PKCE
  (dashed grey lines). The two SPAs are registered as separate service
  providers — `bank-payment-client-spa` and `bank-payment-approver-spa`
  — so a customer token cannot speak admin scopes by replay.
- **WSO2 API Gateway** validates the JWT at the edge (same JWKS the
  Liberty backends use) and routes `/client/v1/*` to `payment-client-svc`,
  `/approver/v1/*` to `payment-approver-svc`. Defense in depth — the
  backends re-validate the same token.
- **The approver service does not own a ledger.** It calls the client
  service's `/v1/internal/transfers/*` endpoints with the admin's own
  Authorization header forwarded (dashed green), so the audit record on
  the client side captures the actual deciding human, not a service
  account.
- **In-memory ledger** for the demo: `zahid` 5005-0001-23 (50,000 BDT)
  and `shaikat` 5005-0004-56 (75,000 BDT). Reserve-on-create + complete-on-approve
  semantics so the visible balance doesn't show "spent twice" during
  pending review.

## Deploy path — pure GitOps

Built today across **four merged platform-gitops MRs**, in order. No
`oc apply` touched either cluster.

| MR | Scope | What it carried |
| :-: | --- | --- |
| **!81** | tenant scaffold + Argo Application | `clusters/spoke-dc-v6/tenants/bank-payment/` (namespace, LimitRange, ResourceQuota, NetworkPolicy set, tenant RBAC) and `clusters/hub-dc-v6/gitops-control/application-bank-payment.yaml` pointing at the tenant repo's `manifests/base/` path |
| **!83** | repo creds | `clusters/spoke-dc-v6/secrets/argocd/externalsecret-gitlab-creds-template.yaml` — a host-wide Argo CD credential template Secret sourced from Vault via ESO. Matches `http://30.30.30.5/*`, so every future tenant repo onboards without per-repo credentialing |
| **!85** | workload RBAC | `clusters/spoke-dc-v6/tenants/bank-payment/argocd-workload-rbac.yaml` — Role granting the spoke Argo SA write verbs on Services, Deployments, ConfigMaps, etc., in the tenant namespace. The default `argocd-tenant-manager` Role only covers SAs + Secrets |
| **!87** *(images path, no platform MR)* | container images | Built locally with `buildah`, pushed to `app-registry.apps.sub.comptech-lab.com/bank-payment/{payment-client-svc,payment-approver-svc,client-frontend,approver-frontend}:latest`. Argo's `selfHeal=true` retry then pulled them and rolled out the workloads |

The build phase took the only out-of-GitOps actions — images are the
supply chain, not the runtime config.

## Three surprises caught during smoke test

1. **`ComparisonError: authentication required`** — the spoke Argo
   couldn't pull the new private repo. The existing
   `gitlab-platform-gitops` Secret only credentials
   `openshift-platform-gitops`. Fix in MR !83 — a creds-template
   matching the whole GitLab host, sourced via ESO so the PAT never
   lands in git.

2. **`services "client-frontend" is forbidden`** — the default tenant
   `argocd-tenant-manager` Role grants ServiceAccount + Secret writes
   but not Deployments / Services. The platform expected workloads to
   land via the per-division `app-tenants` AppSet (which carries
   broader RBAC), and `bank-payment` is a standalone
   `project=default` Application instead. Fix in MR !85 — additive Role
   scoped to this tenant only.

3. **`features.sh` 429-throttled by Maven Central** — the kernel-slim
   Liberty base image installs features at build time from
   `repo.maven.apache.org`, which rate-limited the build node. Switched
   to the **full** Liberty image
   (`docker.io/library/open-liberty:full-java17-openj9`) so features
   are baked in.

The OpenAPI files in `wso2/api-definitions/` carry the `x-wso2-*`
annotations the API Manager needs (CORS, backend URL, security scheme).
Apply them via the APIM publisher REST API or the publisher UI.

## What's intentionally not built yet

- **Persistence**: the ledger is in-memory. Restarts wipe history. JDBC
  + PostgreSQL is the obvious next step; the model is already split
  into `Account` and `Transfer` so the swap is mechanical.
- **mTLS via Service Mesh**: the [demo scope email](/brac-poc/demo-scope-email)
  asks for ambient-mode mTLS and a service-mesh-driven canary. Adding
  the namespace to the mesh + a VirtualService is the next sprint —
  the apps don't need code changes to participate.
- **Audit pipeline**: today the approver svc's decisions land in the
  in-memory ledger only. Real-world: emit a Kafka event per decision
  for downstream reconciliation. The `Observability + Kafka — High
  Priority` line in the scope email points squarely at this gap.
