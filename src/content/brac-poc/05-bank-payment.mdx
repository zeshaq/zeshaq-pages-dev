## What it is

Four apps in one tenant namespace (`bank-payment`) on **spoke-dc-v6**:

| App | Tech | Purpose | Users |
| --- | --- | --- | --- |
| `payment-client-svc` | Open Liberty 26 / JAX-RS / MP-JWT 2.1 | Customer-facing payment API. System of record for accounts + transfers. | `zahid`, `shaikat` |
| `payment-approver-svc` | Open Liberty 26 / MP REST Client | Admin-only API. Pending queue, approve / decline, audit. Calls the client svc's internal endpoints with the admin's own token forwarded. | `admin` |
| `client-frontend` | React 18 + Vite + Tailwind + `react-oidc-context` | Customer SPA. Balance, send-payment form, transfer history. | `zahid`, `shaikat` |
| `approver-frontend` | React 18 SPA | Admin SPA. Pending queue with inline Approve / Decline; audit log. | `admin` |

Tenant: `bank-payment` on `is.apps.sub.comptech-lab.com`. Three users
(`zahid` / `shaikat` / `admin`) with one shared password
(`Hkj38djf&&&`) — `zahid` and `shaikat` carry an `accountNumber` plus
a seed balance; `admin` doesn't own an account, only the approver
role (which the backends derive from `username == "admin"` when the
JWT has no `groups` claim).

**Reading the diagram:**

- **WSO2 IS** issues a JWT to either SPA via OIDC authorization-code +
  PKCE (dashed grey). The two SPAs are registered as separate service
  providers — `bank-payment-client-spa` and `bank-payment-approver-spa`
  — so a customer token's audience doesn't match a call to the admin
  API by accident. (Liberty trusts either audience because the
  approver service forwards admin tokens down to client-svc.)
- **No API Gateway in front of the backends in this build.** Each SPA
  calls its matching Liberty service directly over CORS; the
  `<cors>` block in `server.xml` lets the SPA origin call `/v1/*` with
  `Authorization: Bearer <id_token>`. WSO2 IS issues opaque access
  tokens for public clients, so the JWT-validating backends consume
  the **id_token** as the bearer.
- **The approver service does not own a ledger.** It calls
  `payment-client-svc`'s `/v1/internal/transfers/*` endpoints with the
  admin's own `Authorization` header forwarded (dashed green), so the
  audit record on the client side captures the actual deciding human,
  not a service account.
- **In-memory ledger** for the demo: `zahid` 5005-0001-23 (50,000 BDT)
  and `shaikat` 5005-0004-56 (75,000 BDT). Reserve-on-create +
  complete-on-approve semantics so the visible balance never shows
  "spent twice" during pending review.

## How a payment moves

1. `zahid` opens `payments.apps.spoke-dc-v6.sub.comptech-lab.com`,
   gets bounced to WSO2 IS, logs in, lands back on the SPA with an
   id_token + refresh_token persisted in `localStorage`.
2. He picks `shaikat` from the recipient dropdown (the SPA fetches
   the account directory from `/v1/accounts` with balances redacted
   for non-admins), enters an amount, hits **Send**.
3. SPA `POST /v1/transfers` to `client-api.*` with the Bearer.
   Liberty's mpJwt validates the JWT against the WSO2 JWKS, calls
   `Ledger.createTransfer` — funds are reserved out of `zahid`'s
   visible balance, transfer goes to `PENDING`.
4. `admin` opens `approver.apps.spoke-dc-v6.sub.comptech-lab.com`,
   does the same OIDC dance against the approver SPA. The dashboard
   polls `approver-api.*/v1/pending` every 5 seconds; approver-svc
   forwards the admin's Bearer to `client-api.*/v1/internal/transfers/pending`
   and shows the queue.
5. Admin clicks **Approve**. Approver-svc forwards to
   `client-svc /v1/internal/transfers/{id}/approve`, which moves the
   reserved funds into `shaikat`'s balance and marks the transfer
   `APPROVED` with the admin's username + timestamp on the audit
   trail.

`Decline` is the same path but refunds the reserved amount to the
sender. Either decision is one-shot — the API rejects with `409` if
the transfer is already settled.

## JWT plumbing

A few non-obvious things in the Liberty config worth flagging:

- **Audiences** must be hardcoded in `server.xml` as
  `audiences="bank-payment-client-spa,bank-payment-approver-spa"`. Using
  `${WSO2_JWT_AUDIENCE}` with a comma-separated env value gets parsed
  as a single literal string instead of two trusted audiences, and the
  admin's token (issued for the approver SPA) gets rejected by
  client-svc.
- **`userNameAttribute="email"`** maps `JsonWebToken.getName()` to the
  `email` claim. The `CurrentUser` helper then strips the `@domain`
  part so the in-memory `Ledger` lookup by short username works
  (`zahid@bank-payment.local` → `zahid`).
- **`**` role binding.** `web.xml`'s `<auth-constraint><role-name>**</role-name></auth-constraint>`
  on `/v1/*` does **not** work out of the box under Liberty
  `appSecurity-5.0` — `**` is treated as a literal role name and no
  JWT user is mapped to it, so every request 403s. The fix is an
  `<application-bnd>` block in `server.xml`:

  ```xml
  <application-bnd>
      <security-role name="**">
          <special-subject type="ALL_AUTHENTICATED_USERS"/>
      </security-role>
  </application-bnd>
  ```

- **OPTIONS exemption.** The `<security-constraint>` carries
  `<http-method-omission>OPTIONS</http-method-omission>` so CORS
  preflight requests aren't 401'd before the `<cors>` handler can
  respond.

## TLS

- **Let's Encrypt wildcard** for `*.apps.spoke-dc-v6.sub.comptech-lab.com`
  via cert-manager + the DNS-01 (RFC2136) solver, with TSIG updates
  against the lab's PowerDNS. The spoke IngressController's
  `defaultCertificate` points at the issued Secret, so every Route
  under the spoke ingress gets a browser-trusted cert with no
  per-Route change.
- **Lab CA** is keytool-imported into each Liberty pod's `cacerts` so
  mpJwt's outbound JWKS fetch to `is.apps.sub.comptech-lab.com`
  validates the chain. Liberty's `<ssl>` block points
  `trustStoreRef` at `$JAVA_HOME/lib/security/cacerts` rather than
  Liberty's own empty `key.p12`.

on Nexus.

## What's intentionally not built yet

- **Persistence.** The ledger is in-memory. Restarts wipe history, and
  client-svc runs at `replicas: 1` to avoid split-brain reads. JDBC +
  PostgreSQL (or just a PVC-backed JSON snapshot) is the obvious next
  step; the model is already split into `Account` and `Transfer` so
  the swap is mechanical.
- **API Gateway.** The SPAs call backends directly. Putting WSO2 APIM
  in front is the next sprint — the OpenAPI specs in `wso2/api-definitions/`
  already carry the `x-wso2-*` annotations APIM expects.
- **mTLS via Service Mesh.** The [scope email](/brac-poc/demo-scope-email)
  asks for ambient-mode mTLS and a service-mesh-driven canary. Adding
  the namespace to the mesh + a VirtualService is mechanical — the
  apps don't need code changes to participate.
- **Audit pipeline.** Today the approver svc's decisions land in the
  in-memory ledger only. Real-world: emit a Kafka event per decision
  for downstream reconciliation. That's the `Observability + Kafka
  — High Priority` line in the scope email.
