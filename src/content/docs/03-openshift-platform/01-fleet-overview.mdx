The active OpenShift fleet is two clusters: `hub-dc-v6` — a compact three-master VM management cluster — and `spoke-dc-v6` — three VM masters with three physical workers and Red Hat OpenShift Data Foundation. They run alongside a small set of supporting VMs that provide DNS, image mirror, secrets, and object storage. Everything on the clusters is reconciled from one internal GitLab repository, `comptech-platform/openshift-ops/openshift-platform-gitops`, through the Red Hat ACM plus OpenShift GitOps pull model. This page is the map for the rest of section 3.

## The control flow, in words

1. A platform admin opens a merge request against `platform-gitops` (the canonical clone lives at `/home/ze/platform-gitops` for personal use and `/home/ze/ops-workspace/clones/platform-gitops/` for automation work). The repo has one directory per cluster: `clusters/hub-dc-v6/` and `clusters/spoke-dc-v6/`. There is also a `clusters/hub-dc-v6/gitops-control/` directory that owns the Argo CD AppProjects, the cluster-admin binding, and the ApplicationSet definition.
2. On `hub-dc-v6`, the OLM-installed OpenShift GitOps operator runs an Argo CD instance in the `openshift-gitops` namespace. Its `hub-dc-v6-bootstrap` Application reconciles `clusters/hub-dc-v6/bootstrap/` and `gitops-control/`, then everything else listed in the cluster's root `kustomization.yaml`. That instance is bound to `cluster-admin` per [ADR 0019](/docs/06-architecture-decisions/) so it can install operators and write cluster-scope resources on the hub itself.
3. The hub also installs **ACM** (MultiClusterHub) and **MCE** (Multicluster Engine). MCE provides the registration controllers; ACM adds policy, governance, applications, and console. The hub manifests live under `clusters/hub-dc-v6/platform/acm/` and `clusters/hub-dc-v6/platform/fleet-registration/`.
4. ACM `ManagedCluster/spoke-dc-v6`, `KlusterletAddonConfig`, and a `Placement` (selecting `env=dc, role=spoke, gitops-managed=true`) wire the spoke into the hub. A `GitOpsCluster` resource with `gitopsAddon.enabled: true` tells ACM to also install OpenShift GitOps on the spoke — as a Deployment (the "gitops-addon" install pattern), not as an OLM CSV.
5. An `ApplicationSet` named `spoke-cluster-config-pull` reads the placement decision (via a `clusterDecisionResource` generator and the `acm-placement` ConfigMap) and emits one `Application` per matched spoke. The Applications are labeled `apps.open-cluster-management.io/pull-to-ocm-managed-cluster=true` and annotated `argocd.argoproj.io/skip-reconcile=true` — the hub Argo CD does not reconcile them; ACM wraps each one as `ManifestWork` and ships it to the spoke.
6. On `spoke-dc-v6`, the **klusterlet** (two pods: a registration agent and a work agent) initiated the original outbound connection to the hub and continues to pull `ManifestWork`. The work agent applies the wrapped `Application` resource into the spoke's `openshift-gitops` (technically `acm-openshift-gitops`) namespace.
7. The **spoke Argo CD** reconciles `clusters/spoke-dc-v6/` from internal GitLab against `destination.server: https://kubernetes.default.svc` — that is, against the local API. It installs operators (LSO, ODF, ESO, RHACS, OADP, cert-manager, Loki, Tempo, OpenTelemetry, Tekton, Quay, etc.), then operands (`LVMCluster`, `StorageCluster`, `ClusterSecretStore`, `Central`/`SecuredCluster`, `DataProtectionApplication`, …) per sync-wave ordering.
8. Images for everything come from Nexus. Cluster-side `ImageDigestMirrorSet` / `ImageTagMirrorSet` resources rewrite upstream registry references at node runtime so any pod that says `registry.redhat.io/foo@sha256:...` actually pulls from `mirror-registry.apps.sub.comptech-lab.com/foo@sha256:...`. The default OperatorHub sources are disabled; only the mirrored `cs-redhat-operator-index-v4-20` and `cs-certified-operator-index-v4-20` catalog sources are active.

This is exactly the Red Hat reference architecture for ACM + OpenShift GitOps Basic pull (ADR 0018). Several things that look unusual at first glance — the spoke Argo running as a Deployment, the ApplicationSet template emitting `project: default`, hub Argo bound to `cluster-admin`, the hub having no `StorageClass` — are intentional features of the pattern, not drift.
