OADP has two completely separate paths for protecting volume data, and a single `Backup` can use either or both. This page is the orientation for which path you pick and why, the VolumeSnapshotClass wiring per CSI driver in the lab, and the moving parts to verify.

## CSI snapshots vs. Kopia filesystem backup

| Property | CSI snapshot path | Kopia (node-agent) path |
| --- | --- | --- |
| Source of truth | CSI driver's volume snapshot API | File-level read of the mounted volume |
| Granularity | Block-level (entire volume) | File-level (only paths that exist) |
| Storage of the data | Stays in the CSI provisioner (LVMS, Ceph, ODF) | Pushed to the MinIO bucket as a Kopia repo |
| Restore speed | Fast (clone snapshot) | Slow (re-write files) |
| Cross-cluster restore | Hard (snapshot is provider-local) | Easy (Kopia repo is portable across clusters) |
| Backup cost | Cheap initially, accrues with snapshot retention | Higher per backup, scales with file churn |
| Requirements | CSI driver supports snapshots; matching VolumeSnapshotClass | node-agent DaemonSet healthy on every node |

Rule of thumb in this lab:

- **App PVCs with a few large files (databases, object stores).** CSI snapshot. Kopia file-level pacing would be slow and would not give consistency.
- **App PVCs with many small files (logs, code, build outputs).** Kopia. Filesystem dedup wins and the data ends up in the same MinIO bucket as the backup metadata.
- **Cross-cluster DR.** Kopia, every time. CSI snapshots are not transportable; you cannot restore a Ceph snapshot onto an LVMS cluster.

OADP is configured to do both: every Backup that hits a snapshottable PVC gets a CSI snapshot, *and* the node-agent runs filesystem backup for any volume that is referenced through `defaultVolumesToFsBackup` or marked individually.

## End-to-end flow

<Whiteboard client:load data={{ nodes: snapNodes, edges: snapEdges }} height={720} />

Reading the diagram:

- The user (or a Schedule) files a `Backup` CR.
- The Velero controller fan-outs: the `csi` plugin (an init/controller image inside the Velero pod) creates a `VolumeSnapshot` in the application namespace; the node-agent DaemonSet handles file-level backup for PVCs flagged for FS backup.
- The `VolumeSnapshot` selects a `VolumeSnapshotClass` (per CSI driver) which in turn drives the CSI driver to take a block-level snapshot of the underlying volume.
- Velero writes the backup *metadata* tarball (the CR snapshots, the cluster-scoped resources, the pointers to the snapshots) to MinIO.
- The node-agent's Kopia repository lives inside the same MinIO bucket but under a different key prefix.

The CSI snapshots themselves do not get uploaded to MinIO. They stay in the storage backend. Only their *handles* are written into the Backup metadata.

## VolumeSnapshotClasses in the lab

Two clusters in the fleet have multiple CSI drivers; each driver needs its own VolumeSnapshotClass:

| Cluster | CSI drivers present | VolumeSnapshotClasses |
| --- | --- | --- |
| `hub-dc-v6` | LVMS | `lvms-vg1` |
| `spoke-dc-v6` | LVMS, ODF Ceph RBD, ODF CephFS | `lvms-vg1`, `ocs-storagecluster-rbdplugin-snapclass`, `ocs-storagecluster-cephfsplugin-snapclass` |
| `hub-dr-v6` | LVMS | `lvms-vg1` |
| `spoke-dr-v6` | LVMS, ODF Ceph RBD, ODF CephFS | three (same as spoke-dc-v6) |

`hub-dc-v6` and `hub-dr-v6` are "storage-light" hubs that intentionally do not run ODF. They use LVMS for the small local-disk needs of ACM, RHACM observability, and other hub services. CSI snapshots are still available on LVMS.

A typical VolumeSnapshotClass for LVMS:

```yaml
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshotClass
metadata:
  name: lvms-vg1
  labels:
    velero.io/csi-volumesnapshot-class: "true"   # OADP marker label
driverName: topolvm.io
deletionPolicy: Retain                            # see note below
```

For ODF Ceph RBD, the equivalent uses `driverName: openshift-storage.rbd.csi.ceph.com` and is shipped by the ODF operator. For CephFS, `openshift-storage.cephfs.csi.ceph.com`.

`deletionPolicy: Retain` is set because:

- Velero deletes the `VolumeSnapshot` CR when a Backup expires, but the CSI driver should keep the underlying snapshot if it is still pointed at by a Backup metadata record on MinIO.
- Velero handles the actual snapshot lifecycle through the metadata tarball; `Retain` removes the race where deleting the CR would orphan the metadata.

The `velero.io/csi-volumesnapshot-class: "true"` label is the OADP marker. Without it, the `csi` plugin will not pick up that VolumeSnapshotClass when looking for a class to use during a Backup.

## Per-PVC vs. default-all-volumes

A Backup CR's volume policy is set by these spec fields:

| Field | Effect |
| --- | --- |
| `snapshotVolumes: true` | Enable CSI snapshots. Default true if you have the csi plugin loaded. |
| `defaultVolumesToFsBackup: true` | Every PVC included in this Backup also gets a Kopia file-level backup, *unless* annotated to opt out. |
| `defaultVolumesToFsBackup: false` | Only PVCs with the opt-in annotation get a Kopia backup. |

Opt-in / opt-out annotations on the **pod** (not the PVC):

```yaml
apiVersion: v1
kind: Pod
metadata:
  annotations:
    backup.velero.io/backup-volumes: "data,logs"     # opt-in list
    # OR:
    backup.velero.io/backup-volumes-excludes: "cache"
```

In this lab the convention is:

- Platform Backups (run by the Schedule on `openshift-adp` SA): `snapshotVolumes: true`, `defaultVolumesToFsBackup: false`. CSI-snapshot-only for the big, churning volumes; tenants opt into Kopia on the pods that need it.
- ACM cluster-backup on the hub: `snapshotVolumes: true`, `defaultVolumesToFsBackup: false`. Same shape.
- One-shot tenant Backups: tenant chooses, but default templates set Kopia for `data` volumes.

## Pre-backup hooks for application consistency

CSI snapshots are *crash-consistent*, not *application-consistent*. For most stateless apps that is fine. For databases, you want a pre-backup hook that quiesces writes:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-database
  annotations:
    pre.hook.backup.velero.io/container: "db"
    pre.hook.backup.velero.io/command: '["/bin/sh","-c","mysqldump --single-transaction --quick > /backup/dump.sql"]'
    post.hook.backup.velero.io/container: "db"
    post.hook.backup.velero.io/command: '["/bin/sh","-c","rm /backup/dump.sql"]'
```

For CloudNativePG, the operator manages its own backup hooks and the lab convention is to let CNPG handle its own backup (see §14.13 `cnpg-postgres-operator`). Do not double-back-up a CNPG cluster with OADP file-level backup; you get a half-state.

## Lifecycle of a CSI snapshot

The order of operations a Backup goes through when it touches a snapshottable PVC:

1. The user applies a `Backup` CR (`kubectl apply -f backup.yaml`).
2. Velero lists PVCs in the selected namespaces against the Kubernetes API.
3. Velero delegates each snapshottable PVC to the `csi` plugin.
4. The `csi` plugin creates a `VolumeSnapshot` in the application namespace.
5. The Kubernetes API drives the snapshot through the matching `VolumeSnapshotClass`, which calls the appropriate CSI driver.
6. The CSI driver returns once the `VolumeSnapshotContent` is `ReadyToUse`.
7. The snapshot handle bubbles back through the `csi` plugin to Velero, which records it in the backup metadata.
8. Velero writes the backup tarball (CR snapshots + cluster-scoped resources + snapshot handles) to the Backup Storage Location on MinIO.
9. Velero sets `Backup.status.phase = Completed`.

If the snapshot driver is slow (or stuck), Velero will wait up to `DataProtectionApplication.spec.configuration.velero.defaultItemOperationTimeout` (4h in this lab) before failing the Backup. That is intentionally long because real-world ODF Ceph RBD snapshots of a multi-TiB volume can take 10+ minutes.

## Validation

```bash
K=/home/<user>/.kube/configs/spoke-dc-v6.kubeconfig

# What VolumeSnapshotClasses exist and which are marked for OADP?
oc --kubeconfig "$K" get volumesnapshotclass -o yaml | \
  yq '.items[] | {name: .metadata.name, driver: .driverName, deletionPolicy: .deletionPolicy, oadp: .metadata.labels["velero.io/csi-volumesnapshot-class"]}'

# Current VolumeSnapshots and their state
oc --kubeconfig "$K" get volumesnapshot -A

# Current VolumeSnapshotContents
oc --kubeconfig "$K" get volumesnapshotcontent

# Was a snapshot taken as part of a specific backup?
oc --kubeconfig "$K" -n openshift-adp get backup my-backup-name \
  -o json | jq '.status.csiVolumeSnapshotsAttempted, .status.csiVolumeSnapshotsCompleted'

# Node-agent DaemonSet health
oc --kubeconfig "$K" -n openshift-adp get ds node-agent
```

Expected:

- For every CSI driver on the cluster, exactly one VolumeSnapshotClass labelled `velero.io/csi-volumesnapshot-class: "true"`.
- During a Backup that hits PVCs, `volumesnapshot` and `volumesnapshotcontent` rows transition through `Pending → Ready` quickly.
- `Backup.status.csiVolumeSnapshotsCompleted` equals `csiVolumeSnapshotsAttempted` at the end of a successful run.

## Failure modes

| Symptom | Root cause | Fix | Prevention |
| --- | --- | --- | --- |
| Backup `Partially Failed`, `0 / N csi snapshots completed`. | Multiple VolumeSnapshotClasses for the same driver and none labelled. | Add the `velero.io/csi-volumesnapshot-class: "true"` label to exactly one class per driver. | Make the label part of the ODF / LVMS overlay; CI lint catches duplicates. |
| `VolumeSnapshotContent` stuck `Pending`. | CSI driver pods restarting; or the storage backend is out of snapshot quota. | Inspect the driver pod logs in `openshift-storage` (ODF) or `openshift-storage-lvms` (LVMS). | Schedule backups outside CSI driver upgrade windows. |
| Kopia uploads fail with `connection refused`. | Node-agent cannot reach MinIO from the worker. Same NetworkPolicy / routing issue as §13.2. | Add the egress allow rule for the worker subnet to the MinIO port. | Same prevention: NetworkPolicy and lab firewall changes go through one MR. |
| Restore brings the PVC back empty. | The Backup was configured with `snapshotVolumes: false` and `defaultVolumesToFsBackup: false`. | File a new Backup with one of the two enabled. | Default Schedule template has `snapshotVolumes: true`; never use `false`. |
| `cohort hash mismatch` from Kopia. | Two clusters wrote to the same Kopia prefix (mis-prefixed `objectStorage`). | Wipe the offending Kopia repo prefix; recreate; do *not* share Kopia prefixes across clusters. | Strict prefix-per-cluster discipline at DPA time (already enforced by §13.3 IAM policy). |
| Snapshot CR exists but Backup metadata does not point to it. | OADP CSI plugin disabled (`defaultPlugins` missing `csi` or `EnableCSI` flag missing). | Edit the DPA `spec.configuration.velero.defaultPlugins` and `featureFlags`. | Lint the DPA in CI; assert `csi` plugin and `EnableCSI` flag together. |
