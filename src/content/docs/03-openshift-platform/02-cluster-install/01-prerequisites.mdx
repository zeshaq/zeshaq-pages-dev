Agent-based disconnected install runs on `ocp-bootstrap` and KVM, not from a public CDN. Before the install can start, four gates have to pass: an approved allocation table, applied DNS records, a usable mirrored release in Nexus, and a merged pull-secret on the install host. If any one of them is missing the install will fail in a way that is much harder to diagnose than just stopping.

## DNS records

PowerDNS is the authoritative source for everything under `sub.comptech-lab.com`. The records have to be applied and resolvable from the install host *and* from the eventual cluster nodes' DNS resolver before the install starts:

| Record | Target |
| --- | --- |
| `api.hub-dc-v6.sub.comptech-lab.com` A | `30.30.75.10` |
| `api-int.hub-dc-v6.sub.comptech-lab.com` A | `30.30.75.10` |
| `*.apps.hub-dc-v6.sub.comptech-lab.com` A | `30.30.75.11` |
| `hub-dc-v6-master-{0,1,2}.sub.comptech-lab.com` A | `30.30.75.{13,14,15}` |
| `api.spoke-dc-v6.sub.comptech-lab.com` A | `30.30.75.30` |
| `api-int.spoke-dc-v6.sub.comptech-lab.com` A | `30.30.75.30` |
| `*.apps.spoke-dc-v6.sub.comptech-lab.com` A | `30.30.75.31` |
| `spoke-dc-v6-master-{0,1,2}.sub.comptech-lab.com` A | `30.30.75.{33,34,35}` |
| `spoke-dc-v6-worker-{0,1,2}.sub.comptech-lab.com` A | `30.30.75.{36,37,38}` |

Validate from the bootstrap host:

```bash
for c in hub-dc-v6 spoke-dc-v6; do
  dig @30.30.30.53 api.$c.sub.comptech-lab.com +short
  dig @30.30.30.53 api-int.$c.sub.comptech-lab.com +short
  dig @30.30.30.53 foo.apps.$c.sub.comptech-lab.com +short
done
```

Public DNS is not consulted for these records — the lab DNS recursor resolves them locally and forwards anything outside the local zone upstream.

## Mirrored release + operator catalogs in Nexus

The active OpenShift release for both clusters is `4.20.18`. `oc mirror --v2` runs on `oc-mirror.sub.comptech-lab.com` against the ImageSet at `plans/disconnected-rebuild/environments/dc-lab/imageset-config.yaml` and pushes into:

```text
docker://mirror-registry.apps.sub.comptech-lab.com
```

Before render, confirm:

```bash
# mirror endpoint healthy
curl -ksSI https://mirror-registry.apps.sub.comptech-lab.com/v2/ | head -1
# 401 Unauthorized is the expected unauthenticated response.

# release is inspectable from ocp-bootstrap with the merged pull-secret
oc adm release info \
  --registry-config /home/ze/ocp-clusters/hub-dc-v6/pull-secret.merged.json \
  $(cat /home/ze/ocp-clusters/hub-dc-v6/release-image)
```

oc-mirror's generated cluster resources land under `/home/ze/ocp-mirror-workspaces/dc-lab/full-workspace/working-dir/cluster-resources/`:

- `idms-oc-mirror.yaml` — `ImageDigestMirrorSet` covering the release/operator/operand image refs
- `itms-oc-mirror.yaml` — `ImageTagMirrorSet` for any tag-pinned refs
- `cs-redhat-operator-index-v4-20.yaml` — `CatalogSource` for the mirrored Red Hat operator catalog
- `cs-certified-operator-index-v4-20.yaml` — `CatalogSource` for the certified catalog
- `clustercatalog-*.yaml` — OLM v1 `ClusterCatalog` equivalents for the same catalogs
- signature ConfigMaps for verified payload pull

The install-config has to embed an `imageDigestSources` block that points at the mirrored release path. Do not paste in `oc mirror`'s output verbatim — review it, then commit the durable IDMS/ITMS/CatalogSource manifests into `platform-gitops` under `clusters/<cluster>/platform/image-mirrors/` and `clusters/<cluster>/platform/catalogs/` (covered in [§6 disconnected image supply](/docs/03-openshift-platform/06-disconnected-image-supply/01-oc-mirror-workspace/)).

## Pull-secret custody

The merged pull-secret combines Red Hat customer pull-secret with the local Nexus mirror-registry credential. It lives **only** under:

```text
ocp-bootstrap:/home/ze/ocp-clusters/<cluster>/pull-secret.merged.json
"$PULL_SECRET"  # workstation-local, git-ignored   # workstation-local, git-ignored
```

It is never committed to GitLab or GitHub. The install-config template references it by placeholder `<INSERT_LOCAL_ONLY_PULL_SECRET_JSON>`; the render step substitutes the actual file content in-memory and writes the final `install-config.yaml` only under `ocp-bootstrap:/home/ze/ocp-clusters/<cluster>/`.

## FIPS / etcd / disk encryption posture

The install posture is fixed per [ADR 0005](/docs/06-architecture-decisions/):

| Knob | Value | Set where |
| --- | --- | --- |
| `fips: true` | Enabled | install-config |
| etcd encryption at rest | Enabled | post-install via `APIServer/cluster` |
| Boot-device disk encryption | TPM2-backed LUKS | `99-master-tpm2-encryption.bu` Butane source, rendered to `openshift/99-master-tpm2-encryption.yaml` |
| IPv6 disabled in host NMState | Yes | `agent-config.yaml` `ipv6.enabled: false` per interface |
| Public node IPs | Not allowed | enforced by network plan; no `br-vm` attachment on cluster VMs |

FIPS mode rejects `ssh-ed25519` keys in the install-config — supply an RSA or ECDSA public key only. This caught a render run during spoke install (issue #93) and is now documented in the spoke `validation.md`.

## IPv6 baseline (OVN-Kubernetes-safe)

[ADR 0005](/docs/06-architecture-decisions/) originally said "disable IPv6 in host networking" — but [ADR 0026](/docs/06-architecture-decisions/) supersedes that for any OVN-Kubernetes cluster. The operative pre-install requirement is now "IPv6 is not used for cluster traffic", defined as four verifiable invariants:

1. `clusterNetwork` and `serviceNetwork` are IPv4-only.
2. No admin-managed IPv6 addresses or routes on physical interfaces (link-local `fe80::/10` on the OVN-K geneve tunnel is permitted; `::1/128 scope host` on `lo` is permitted and required).
3. No upstream DHCPv6 / Router-Advertisement listeners on the `30.30.0.0/16` fabric.
4. No application workload binds to an IPv6 podIP.

The kernel-arg `ipv6.disable=1` MachineConfig and the `sysctl disable_ipv6=1` MachineConfig variant are both forbidden because OVN-Kubernetes's `ovnkube-controller` startup runs `sysctl -w net.ipv6.conf.all.forwarding=0` unconditionally and crash-loops if `/proc/sys/net/ipv6/conf/all/forwarding` is unwritable (incident #135, MRs `!2` and `!4`).

The `agent-config.yaml` per-interface `ipv6.enabled: false` setting satisfies the "no admin-managed IPv6 on physical interfaces" invariant without disabling the kernel module.

## Bootstrap and rendezvous host

Agent-based install needs one rendezvous host. For the compact hub the rendezvous is `hub-dc-v6-master-0` at `30.30.75.13`; for the spoke it is `spoke-dc-v6-master-0` at `30.30.75.33`. The "bootstrap" allocation row (`.12` for the hub, `.32` for the spoke) is reserved for method flexibility but is **not provisioned** for the agent-based flow — the installer uses one of the masters as both rendezvous and temporary bootstrap until handoff.

`ocp-bootstrap` (separate Linux VM, lab `/24`) is the operator's render host. It holds:

- the merged pull-secret
- `~/ocp-clusters/<cluster>/install-config.yaml` (final, rendered, never committed)
- `~/ocp-clusters/<cluster>/agent-config.yaml`
- `~/ocp-clusters/<cluster>/openshift/99-{master,worker}-tpm2-encryption.yaml`
- `~/ocp-clusters/<cluster>/<cluster>-agent.iso` after `openshift-install agent create image`
- `~/ocp-clusters/<cluster>/auth/kubeconfig` and `kubeadmin-password` after install — these never leave the host's `auth/` directory uncopied; the copies that *do* leave are deliberate and tracked under [§4 platform-admin-handoff](/docs/03-openshift-platform/04-acm-multicluster/02-managedcluster-registration/).

## The render-and-stop helpers

Two repeatable scripts in `opp-full-plat` enforce the gates:

```bash
# Hub:
cd /home/ze/opp-full-plat
scripts/rebuild/openshift-install/prepare-hub-dc-v6.sh
scripts/rebuild/openshift-install/create-hub-dc-v6-vm-definitions.sh

# Spoke:
scripts/rebuild/openshift-install/prepare-spoke-dc-v6.sh
scripts/rebuild/openshift-install/create-spoke-dc-v6-vm-definitions.sh
```

`prepare-*` stages mirror auth, pulls reviewed inputs, renders `install-config.yaml` / `agent-config.yaml` / TPM2 MachineConfigs on `ocp-bootstrap`. `create-*-vm-definitions.sh` defines the libvirt domains on the KVM host (deterministic MACs from the allocation table) and stops before VM boot. VM boot and physical worker boot are separate explicit approval gates.

## Stop-before-install criteria

Stop immediately and re-check if any of the following is true:

- a VM gets the wrong IP, MAC, gateway, DNS server, or hostname
- IPv6 routing appears unexpectedly on a cluster interface
- any image pull goes to the public internet
- FIPS, etcd encryption, or disk encryption deviates from plan
- generated files expose secrets in tracked paths
- DNS resolution for any `api*` / `*.apps.*` record returns the wrong VIP
- `oc adm release info` against the mirrored release fails from `ocp-bootstrap`
