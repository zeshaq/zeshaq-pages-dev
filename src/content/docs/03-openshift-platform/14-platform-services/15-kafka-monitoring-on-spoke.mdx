The lab Kafka cluster runs on three VMs outside OpenShift — `kafka-0/1/2.sub.comptech-lab.com` on `30.30.30.24/25/26`, Kafka 4.2.0 KRaft on OpenJDK 21. Monitoring lives on `spoke-dc-v6` so the metrics surface stays adjacent to the workloads that produce against Kafka. This page is the Phase 1 shape — the namespace, the danielqsj `kafka-exporter` Deployment, the JMX broker-side javaagent install, the OTel fanout collector, and the open defect under issue #269 that still has `KafkaExporterDown` firing.

## Why on spoke and not on the brokers

Two reasons:

- **Consumer-group lag and cluster-shape metrics come from a Kafka admin client, not from JMX.** danielqsj/kafka-exporter speaks the Kafka protocol, asks the cluster for topic/partition/group metadata, and exposes Prometheus metrics. Running that next to the brokers wastes a VM; running it on the cluster gives it the same lifecycle as everything else (Argo, RBAC, NetworkPolicy, image supply via Nexus).
- **JMX is per-JVM.** Broker internals (`UnderReplicatedPartitions`, `ActiveControllerCount`, raft-channel rates) need a `jmx_prometheus_javaagent` loaded into each broker's JVM. The agent is on the VMs; the scrape target is `30.30.30.24-26:9404` reached over the cluster's egress NetworkPolicy.

UFW on each broker allows `30.30.0.0/16 → :9404` only — the cluster's pod CIDR ranges fit inside `30.30.0.0/16`, no broader allow.

Reading the diagram:

- `kafka-exporter` in the `kafka-monitoring` namespace dials `kafka-0/1/2:9092` over PLAINTEXT and exposes consumer-group/topic/partition metrics on `:9308`.
- A `ServiceMonitor` plus a `PrometheusRule` (`kafka.rules`) wire the exporter into user-workload monitoring; alerts come from this path.
- A second OTel `OpenTelemetryCollector` (`kafka-fanout`) prometheus-scrapes the same exporter and tees traces + metrics + logs to **SigNoz** (VM at `30.30.30.33:4317`) and traces also to the in-cluster TempoStack via `otelcol-platform-collector`.

## Broker-side JMX javaagent

The agent is `jmx_prometheus_javaagent-1.0.1.jar`, pinned by SHA-256 `7d61f737fd661610ccc14aea79764faa1ea94a340cbc8f0029b3d2edea3d80c1`. It is sourced from Maven Central. Maven Central does not publish a `.sha256` sibling for this artifact — we verify the download against the published `.sha1` (`f9c53eb0aa1828c3c9cd7647238563e43a4ed68c`) and `.md5` (`f525facee9d400d163e5366c02d3aa68`), then record the SHA-256 for our records and re-verify on each broker after copy.

Files placed on each broker (root:root):

```text
/opt/jmx-exporter/jmx_prometheus_javaagent.jar              (0644)
/opt/jmx-exporter/kafka-jmx-config.yml                      (0644)
/etc/systemd/system/kafka.service.d/10-jmx-exporter.conf    (0644)
```

The systemd drop-in (identical on all three brokers):

```ini
[Service]
Environment="KAFKA_OPTS=-javaagent:/opt/jmx-exporter/jmx_prometheus_javaagent.jar=9404:/opt/jmx-exporter/kafka-jmx-config.yml"
```

`server.properties`, `log4j2.yaml`, and the pre-existing `KAFKA_HEAP_OPTS` / `LOG_DIR` / `KAFKA_LOG4J_OPTS` environment are **not touched**. The rolling restart is one broker at a time with URP=0 verified between each. Each broker heals in &lt;15s; KRaft leadership transitions are expected during the restart of the current leader.

### The 17-rule JMX config

The config has 17 rule blocks (well under the ~25 cap). Rules are ordered specific-first so per-topic/per-partition matches take precedence over the generic catch-all:

| # | Block | What it captures |
| --- | --- | --- |
| 1 | per-topic per-partition `<>Value` | gauges for partition offsets, ISR shape |
| 2 | per-topic broker `<>Count` | message-rate counters |
| 3 | per-broker replica fetcher | follower lag metrics |
| 4 | KRaft raft `*-total` / `*-rate` | quorum + log-replication counters |
| 5 | KRaft raft other attrs | gauges for state |
| 6 | KRaft raft channel | raft-channel rate/throughput |
| 7 | per-user/per-client quota | request quota counters |
| 8 | network processor connection | connection rate, idle |
| 9 | request metrics counters | per-API call counts |
| 10 | request metrics percentiles | latency p50/p95/p99 |
| 11 | generic `<>Count` | catch-all counters |
| 12 | generic `<>Percentile` | catch-all gauges |
| 13 | generic `<>Value` | covers `ReplicaManager,UnderReplicatedPartitions` and `KafkaController,ActiveControllerCount` |
| 14 | JVM heap/non-heap | memory pools |
| 15 | JVM GC count/time | GC pressure |
| 16 | JVM thread counters | thread pool size |
| 17 | tail catch-all `kafka.*` → `kafka_generic_*` UNTYPED | so unknown beans are not blackholed |

Each broker emits ~10k `kafka_` metric lines per scrape; ~40% of the volume is the tail catch-all. Scrape latency from `localhost:9404/metrics` is ~600ms, response ~1.34 MB. If fleet-scale ingestion becomes a problem, tighten the tail at scrape time via `metricRelabelConfigs` `drop` actions on the ServiceMonitor rather than re-rolling the broker config.

## The OTel fanout collector

`kafka-fanout` is a second `OpenTelemetryCollector` (mode `Deployment`, 1 replica) that exists for one reason — the lab wants every Kafka signal in **SigNoz**, not just Prometheus metrics. The collector has three pipelines:

- **traces** — OTLP-in from `bank-employees-jboss-chat` producers/consumers → SigNoz + Tempo (tee'd, so existing Tempo workflows keep working).
- **metrics** — OTLP-in and a built-in `prometheus` receiver that scrapes the danielqsj exporter on `kafka-exporter.kafka-monitoring.svc:9308` → SigNoz.
- **logs** — OTLP-in → SigNoz.

The prometheus receiver also has a `kafka-broker-jmx` job staged for Phase 2, scraping `kafka-0/1/2.sub.comptech-lab.com:9404`. The job is in the config; whether it actually scrapes depends on Phase 2 (see §Phase 2 below).

Sampling on the trace pipeline is the same tail-sampling shape as the bank-payment fanout — keep every error, keep every span over 500ms, sample 10% of the rest:

```yaml
tail_sampling:
  decision_wait: 10s
  num_traces: 5000
  policies:
    - { name: keep-errors, type: status_code, status_code: { status_codes: [ERROR] } }
    - { name: keep-slow,   type: latency,     latency:     { threshold_ms: 500 } }
    - { name: sample-rest, type: probabilistic, probabilistic: { sampling_percentage: 10 } }
```

Resource attributes are upserted onto every signal: `deployment.environment=lab`, `service.namespace=kafka`, `k8s.cluster.name=spoke-dc-v6`. SigNoz's Services view groups brokers, jboss-chat producer, and jboss-chat consumer under one logical "kafka" surface.

## The open defect — issue #269

At 2026-05-12 closeout the validation pass found a true-positive `KafkaExporterDown`. Root cause is two-part:

1. **Namespace was mislabeled `openshift.io/cluster-monitoring: "true"`.** OCP routes any ServiceMonitor in that namespace to the **platform** Prometheus (`openshift-monitoring/k8s`), not UWM. Effect: UWM has 0 active targets; platform Prom claims the ServiceMonitor.
2. **NetworkPolicy ingress only allows UWM.** `allow-ingress-from-user-workload-monitoring` allows ingress from `openshift-user-workload-monitoring` only — not from `openshift-monitoring`. The platform scrape then hits the default-deny and times out with `context deadline exceeded`. `up=0` → `KafkaExporterDown` fires true-positive.

The fix (preferred): **drop the mislabel.** The namespace description in `namespace.yaml` already says "Scraped by the openshift-user-workload-monitoring Prometheus" — the label was a copy-paste from a hub-side namespace. The MR removes the label only; the NetworkPolicy stays as-is because UWM is the intended scraper.

Alternative (rejected): expand the ingress NetworkPolicy to also allow `openshift-monitoring`. Rejected because it would silently route the ServiceMonitor to the wrong Prometheus and the alert would clear without anyone noticing the platform/UWM split was wrong.

## Phase 2 — broker JMX scrape via UWM

Tracked under issue #273. The blocker is that the spoke's `argocd-cm` `resource.exclusions` excludes **both** `core/Endpoints` **and** `discovery.k8s.io/EndpointSlice` (non-default; see [the spoke argo exclusions reference](../05-gitops-operating-model)). The external-IP scrape target needs one of those representations and Argo will not sync either today.

The two options Phase 2 picks between:

- Relax the exclusion list to allow `EndpointSlice` (smaller blast radius than re-enabling `Endpoints`).
- Pick a representation Argo will sync — a `Probe` CR pointing at the JMX endpoints, or a selectorless Service + an externally managed EndpointSlice.

Broker-side javaagents are installed and idle. UFW `:9404` ingress is in place. The cluster cannot scrape them until #273 lands.

## Failure modes

| Symptom | Root cause | Fix |
| --- | --- | --- |
| `KafkaExporterDown` firing despite the pod being Running. | Namespace mislabeled `openshift.io/cluster-monitoring=true` and platform Prom can't reach the pod. | Drop the label; UWM picks the ServiceMonitor up correctly. |
| Exporter pod `ImagePullBackOff`. | `app-registry-pull` Secret not materialised yet — ESO still reconciling. | `oc -n kafka-monitoring get externalsecret app-registry-pull` should show `SecretSynced Ready=True`. |
| Exporter Running but no `kafka_brokers` series. | The exporter can't reach `kafka-0/1/2:9092` from the worker. | Check `allow-egress-kafka-vms` NetworkPolicy; verify UFW on the brokers allows `30.30.0.0/16`. |
| JMX scrape from a worker pod times out. | Broker-side UFW doesn't allow the source IP. | UFW must allow `30.30.0.0/16` (not just specific worker IPs). |
| `kafka-fanout` collector restarting. | Tail-sampling memory pressure under high span volume. | Raise `num_traces` ceiling or shorten `decision_wait`; or scale to 2 replicas. |
