The `platform-services` overlay on `spoke-dc-v6` is the set of operators and operand CRs that sit between the cluster baseline (storage, networking, ACM, GitOps, ESO) and the tenant application namespaces. Everything in `clusters/spoke-dc-v6/platform-services/` of the platform-gitops repo. Together, they are the platform contract a tenant app inherits when it onboards.

This page is two things: an orientation diagram for the whole overlay, and the cert-manager deep-dive (the file name on disk is `01-cert-manager.mdx`, which is also the natural lead because cert-manager underwrites most of the other components). The remaining 13 pages each cover a single component or pair.

## cert-manager — why it is foundational

cert-manager runs as a single operator in `cert-manager` namespace, installed via the Red Hat cert-manager Operator subscription. The operand is `OperatorConfig` plus a `ClusterIssuer` (or two), and then every other namespace files `Certificate` CRs as needed.

In the lab, the spoke runs two ClusterIssuers:

| ClusterIssuer | Purpose | Issuance mode |
| --- | --- | --- |
| `lab-step-ca` | Internal `*.apps.sub.comptech-lab.com` certs | ACME via the lab smallstep CA, HTTP-01 challenges resolved by the OCP Router |
| `selfsigned-bootstrap` | Bootstrap CAs and very short-lived experiments | `selfSigned: {}`; not for production use |

The smallstep CA path is the one almost all platform services use. cert-manager solves an HTTP-01 challenge by adding a temporary `HTTPRoute` (or, for Routes, by patching the Route's TLS config), the smallstep ACME server checks the challenge, and a `Certificate` is issued. The certificate lands as a Secret in the requesting namespace.

A typical `Certificate` resource:

```yaml
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: quay-platform
  namespace: quay-enterprise
spec:
  secretName: quay-platform-tls          # the Secret cert-manager creates
  duration: 2160h                         # 90 days
  renewBefore: 360h                       # rotate 15 days before expiry
  issuerRef:
    name: lab-step-ca
    kind: ClusterIssuer
    group: cert-manager.io
  commonName: quay.apps.sub.comptech-lab.com
  dnsNames:
    - quay.apps.sub.comptech-lab.com
  privateKey:
    algorithm: ECDSA
    size: 256
```

Notes:

- The Secret name (`spec.secretName`) is what consumers reference. Routes mount it via `route.tls.certificateSecretName`; ingresses via `tls[].secretName`; operands (Loki, Tempo, Quay) via their own config-bundle pattern.
- `duration: 90d` and `renewBefore: 15d` gives a 75-day-old certificate a two-week renewal window. cert-manager handles the rotation automatically.
- ECDSA-256 is the lab default; it gives short certs and small handshakes.

## How platform components consume certs

Each component does it slightly differently, which is the cost of cert-manager being foundational rather than a piece of any one component:

| Component | Pattern |
| --- | --- |
| Routes (OCP) | `route.spec.tls.externalCertificate.name = "<secret>"` referencing the cert-manager-issued Secret. Available since OCP 4.16 via the `RouteExternalCertificate` feature gate. |
| Ingresses (HTTPRoute / Gateway) | `tls[].secretName` referencing the Secret directly. |
| Quay | A `Certificate` produces a Secret; the Quay operator's `quay-config-bundle-secret` references it via `config.yaml` (or `tls` component is set to `managed: true` and Quay manages its own — see §14.14). |
| LokiStack | The LokiStack operator does not consume cert-manager directly. mTLS between components is managed by the operator. External access uses a Route that does. |
| TempoStack | Same as Loki — operator-managed internal TLS; Route uses cert-manager. |
| Open Liberty | `OpenLibertyApplication.spec.route.certificateSecretName` references the Secret. |

The unifying pattern: cert-manager creates Secrets; downstream resources reference those Secrets. cert-manager does not push certs at consumers; consumers pull them by name.

## Installation shape

The operator is installed via Subscription (sync-wave `5`); the `OperatorConfig` and the two ClusterIssuers follow at sync-wave `10`:

```yaml
apiVersion: operator.openshift.io/v1alpha1
kind: CertManager
metadata:
  name: cluster
spec:
  managementState: Managed
  logLevel: Normal
  operatorLogLevel: Normal
```

```yaml
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: lab-step-ca
spec:
  acme:
    server: https://step-ca.sub.comptech-lab.com/acme/acme/directory
    privateKeySecretRef:
      name: acme-account-key-lab-step-ca
    skipTLSVerify: false
    solvers:
      - http01:
          ingress:
            class: openshift-default
```

The HTTP-01 solver uses the OCP Router's default ingress class because the lab does not run a separate ingress controller. The smallstep CA's root certificate is trusted by every node in the fleet via the cluster's image-registry-config trust bundle — that wiring is in §6 Disconnected Image Supply.

## Failure modes

| Symptom | Root cause | Fix | Prevention |
| --- | --- | --- | --- |
| `Certificate` stuck `Issuing`. | HTTP-01 challenge fails — Router cannot reach `step-ca`, or DNS for the SAN does not resolve. | `oc describe certificate <name>`; check `Order` and `Challenge` CRs in same ns. | Verify lab DNS resolves the SAN to the cluster ingress before requesting. |
| Certificate Secret exists but Route still serves a stale cert. | OCP Route reload was missed; Router pod cached previous TLS. | `oc rollout restart -n openshift-ingress deployment/router-default`. | Use `RouteExternalCertificate` (which the Router watches) over manual TLS blocks. |
| `step-ca` rejects an ACME request with `bad-csr`. | Order's CSR has an unauthorised DNS name. | Validate the `Certificate.spec.dnsNames` against the step-ca's `acme` provisioner's allow list. | Keep step-ca's provisioner config in `vm-platform-ops` so the allow list is reviewable. |
| Bunch of Certificates degrade overnight. | ClusterIssuer rotation failed during a step-ca outage. | `oc get certificaterequest -A` — they'll be `Pending`; restart cert-manager to retry. | Monitor step-ca availability; alert on issuance latency. |
