`ExternalSecret` is the *request* CR — the only thing app teams ever write in §10. CSS and SecretStore are platform concerns; tenants own their `ExternalSecret`s. This page covers the shape, the template DSL, refresh behavior, and `ClusterExternalSecret` for the one fan-out case the platform uses.

## The minimal shape

```yaml
apiVersion: external-secrets.io/v1
kind: ExternalSecret
metadata:
  name: db-password
  namespace: apps-platform-checkout
spec:
  refreshInterval: 1h
  secretStoreRef:
    kind: SecretStore        # or ClusterSecretStore for platform paths
    name: vault-apps         # or vault-platform
  target:
    name: db-password        # the K8s Secret that will appear
  data:
    - secretKey: password
      remoteRef:
        key: apps/platform/checkout/prod/db
        property: password
```

This creates `Secret/db-password` in `apps-platform-checkout`, type `Opaque`, with one key `password` carrying the Vault value at `secret/data/apps/platform/checkout/prod/db` field `password`.

## Fields and what they do

| Field | Meaning |
| --- | --- |
| `spec.refreshInterval` | how often ESO re-reads Vault. `0` = once. Common values: `1h` (regular), `15m` (rapidly-rotating creds), `24h` (slow-moving robot tokens). |
| `spec.secretStoreRef.kind` | `SecretStore` for tenant scope, `ClusterSecretStore` for platform scope. |
| `spec.secretStoreRef.name` | always `vault-apps` (tenant) or `vault-platform` (platform), in this lab. |
| `spec.target.name` | the Kubernetes Secret name. If absent, ESO uses the `metadata.name` of the ExternalSecret. |
| `spec.target.creationPolicy` | `Owner` (default — ESO owns the Secret) / `Orphan` (don't delete on ES delete) / `Merge` (combine with existing Secret) / `None` (don't create, fail if missing). |
| `spec.target.deletionPolicy` | `Retain` / `Delete` / `Merge`. Default `Retain` — deleting the ES does not delete the Secret. |
| `spec.target.template` | optional templating block, see below. |
| `spec.data[]` | direct key/property mapping (no templating). |
| `spec.dataFrom[]` | bulk read — all fields of a kv-v2 entry, or a find-by-tag, become keys. |

`data[]` vs `dataFrom[]`: the first lists keys one-by-one and is the common case; the second pulls every field of a Vault entry as Secret keys (rarely useful, only when the Vault entry happens to have the same field names the consumer expects).

## The template block

The template block lets you produce a Secret in any shape consumers need — including `kubernetes.io/dockerconfigjson` (pull secret), `kubernetes.io/tls` (TLS pair), Liberty `server.xml` snippets, or arbitrary JSON.

```yaml
spec:
  target:
    template:
      type: kubernetes.io/dockerconfigjson
      data:
        .dockerconfigjson: |
          {
            "auths": {
              "quay.apps.sub.comptech-lab.com": {
                "username": "{{ .username }}",
                "password": "{{ .token }}"
              }
            }
          }
  data:
    - secretKey: username
      remoteRef: { key: apps/platform/checkout/ci/quay-robot, property: username }
    - secretKey: token
      remoteRef: { key: apps/platform/checkout/ci/quay-robot, property: token }
```

What's going on:

1. `data[]` pulls the two Vault fields into intermediate keys `username` and `token`.
2. The `template.data` block uses Go-template syntax (`{{ .username }}`) to compose the final `.dockerconfigjson` payload.
3. The resulting Secret has type `kubernetes.io/dockerconfigjson` and a single key `.dockerconfigjson` containing the JSON blob.

The template engine is **Go templates with sprig functions** (`b64enc`, `b64dec`, `quote`, `default`, `lower`, `upper`, `replace`, `trim`, `nindent`, …). Common needs:

- Base64-encode a Vault value before placing it in a Secret data key:
  `{{ .raw | b64enc }}`
- Encode an entire JSON blob and pass through:
  `{{ tpl .json . | b64enc }}`
- Provide a default if Vault returns empty:
  `{{ .key | default "fallback" }}`

## Templated TLS Secret (cert-manager interop)

```yaml
apiVersion: external-secrets.io/v1
kind: ExternalSecret
metadata:
  name: app-tls
  namespace: apps-platform-checkout
spec:
  refreshInterval: 24h
  secretStoreRef:
    kind: SecretStore
    name: vault-apps
  target:
    name: app-tls
    template:
      type: kubernetes.io/tls
      data:
        tls.crt: "{{ .crt }}"
        tls.key: "{{ .key }}"
        ca.crt:  "{{ .ca }}"
  data:
    - secretKey: crt
      remoteRef: { key: apps/platform/checkout/prod/tls, property: cert }
    - secretKey: key
      remoteRef: { key: apps/platform/checkout/prod/tls, property: key }
    - secretKey: ca
      remoteRef: { key: apps/platform/checkout/prod/tls, property: ca }
```

This produces a TLS Secret that an `Ingress`, `Route`, or service-side TLS-loader can consume directly.

## Refresh semantics — what actually happens

When `refreshInterval: 1h` is set:

1. ESO's reconciler loops every `refreshInterval` and re-reads Vault.
2. If the value changed, it updates the target `Secret`. The Secret's `data` is overwritten.
3. Pods that have already mounted the Secret as a *volume* see the new value within the kubelet's atomic-rewrite period (`syncFrequency` × 2; typically &lt;1 minute).
4. Pods that consumed the Secret as **env vars** see nothing — env is set at pod start. The pod must restart (usually triggered by a controller watching the Secret's resourceVersion via Reloader, or by a manual rollout).

A common gotcha is "the secret rotated in Vault but my app is still using the old value." 90% of the time the cause is **env-var consumption** with no Reloader. Best practice in this lab: mount secrets as files, not env vars. The §10 rotation page covers this in depth.

## ClusterExternalSecret — the one fan-out case

`ClusterExternalSecret` (CES) is a cluster-scoped CR that creates `ExternalSecret`s in many namespaces. The platform uses it exactly once: to fan out `Secret/app-registry-pull` (dockerconfigjson for the Nexus app-registry) to every tenant namespace labelled `apps.platform/tenant=true`.

```yaml
apiVersion: external-secrets.io/v1beta1
kind: ClusterExternalSecret
metadata:
  name: app-registry-pull
spec:
  externalSecretName: app-registry-pull
  namespaceSelector:
    matchLabels:
      apps.platform/tenant: "true"
  refreshTime: 1h
  externalSecretSpec:
    secretStoreRef:
      kind: ClusterSecretStore
      name: vault-platform
    target:
      name: app-registry-pull
      template:
        type: kubernetes.io/dockerconfigjson
        data:
          .dockerconfigjson: "{{ .dockerconfigjson }}"
    data:
      - secretKey: dockerconfigjson
        remoteRef:
          key: ocp/<cluster>/registries/app-registry-pull
          property: dockerconfigjson
```

What this does:

1. ESO watches `Namespace` events for label `apps.platform/tenant=true`.
2. For each matching namespace, it materializes an `ExternalSecret/app-registry-pull` that pulls from the platform Vault path and templates a dockerconfigjson Secret.
3. Tenant teams then patch their default ServiceAccount to use the pull Secret:
   ```bash
   oc -n <tenant-ns> patch sa default \
     -p '{"imagePullSecrets":[{"name":"app-registry-pull"}]}'
   ```

Why we use CES here and only here:

- The pull credential is **platform-owned, shared by every tenant**. Per-tenant ExternalSecrets would still all read the same Vault path, with no isolation benefit and 30× the YAML.
- Tenant opt-in is one-label-change rather than copying a manifest into every tenant overlay.
- CES references `ClusterSecretStore vault-platform` — the only path here that goes through platform scope.

Per-app secrets (DB passwords, robot tokens, TLS) always use per-tenant `ExternalSecret`s.

## Conventions we enforce by review

| Convention | Why |
| --- | --- |
| `refreshInterval` set explicitly (never default) | makes rotation behavior reviewable |
| `target.name` set even when it matches the ES name | makes the manifest self-documenting |
| `template:` used whenever the consumer expects anything other than `Opaque` | avoids second-stage transformation |
| Vault paths follow the convention `apps/<division>/<app>/<env>/<key>` for tenant; `ocp/<cluster>/...` for platform | matches the role/policy globs |
| Tenant ExternalSecrets reference `kind: SecretStore` (never `ClusterSecretStore`) | preserves isolation |
| Platform ExternalSecrets in operator namespaces use `kind: ClusterSecretStore` | one cross-cutting CSS for platform |

## Failure modes

| Symptom | Root cause | Fix |
| --- | --- | --- |
| ES `Ready=False not found` | Vault path doesn't exist or has no fields | `vault kv get -format=json secret/<path>` |
| ES `Ready=False property not found` | the kv entry exists but lacks the requested field | inspect with `vault kv get -format=json` |
| ES `Ready=True`, Secret created, but pod doesn't see new value | env-var consumption, no Reloader | mount as file, or add Reloader annotation, or `rollout restart` |
| ES `Ready=True`, Secret type wrong | template type missing or typo | set `target.template.type` explicitly |
| CES doesn't materialize ES in a tenant ns | namespace label missing | `oc label ns <tenant> apps.platform/tenant=true` |
| Templated dockerconfigjson rejected by registry | base64 layer wrong (double-encoded, or single-line vs multi-line) | template the JSON as a string; ESO base64-encodes the data values when writing to Secret |
