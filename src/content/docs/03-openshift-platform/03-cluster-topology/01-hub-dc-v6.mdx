`hub-dc-v6` is the compact management cluster — three masters and no workers. It runs the fleet's control plane: hub Argo CD, ACM MultiClusterHub, MCE, the ApplicationSet that drives the pull model, RHACS Central (with LVMS-backed storage), External Secrets Operator, and cert-manager. It does **not** run ODF, NooBaa, OpenShift Pipelines, or logging/tracing/observability operators ([ADR 0004](/docs/06-architecture-decisions/)).

## Why compact 3-master

A compact cluster is OpenShift's pattern for a control-plane-only cluster where the masters are also workers (they have the `node-role.kubernetes.io/master` and `node-role.kubernetes.io/worker` labels both). It is the right shape for a management hub because:

- the management workload (Argo CD, ACM, MCE, RHACS Central, ESO operator pods) is small enough to fit on three nodes.
- there are no application workloads to isolate from the control plane.
- doubling the cluster to six nodes (3 masters + 3 dedicated workers) would add hardware cost without adding value.

The trade-off: the cluster cannot tolerate two simultaneous master failures (it loses etcd quorum), and the masters carry both the control plane and the management workload load. Both are accepted in this design.

## Storage posture — LVMS only

[ADR 0004](/docs/06-architecture-decisions/) is "Keep Management Hubs Storage-Light":

- **LVMS is installed** on the hub. It provides node-local LVM-backed PersistentVolumes via the TopoLVM CSI driver. The default StorageClass is `lvms-vg1` and is consumed by RHACS Central, RHACS DB, and RHACS Scanner V4 components.
- **ODF / NooBaa is not installed** on the hub. The lessons from the pre-v6 `hub-dr` storage recovery were that hub-local ODF added more operational risk (stale finalizers, NooBaa RBD-PVC dependency, LVMS device-set mistakes) than value. ACM backup/restore uses external MinIO instead.
- **OpenShift Pipelines, logging, tracing, NooBaa, observability stacks are not installed** on the hub. They live on the workload cluster (`spoke-dc-v6`) where ODF backs them.
- **The internal OpenShift image registry is `Removed`** — `Config.imageregistry/cluster.spec.managementState=Removed`, no PV, no SC. The hub does not need to host an in-cluster image registry; workloads pull from Nexus.

`Config.imageregistry/cluster`:

```yaml
apiVersion: imageregistry.operator.openshift.io/v1
kind: Config
metadata:
  name: cluster
spec:
  managementState: Removed
```

This is intentional design, not drift. The memory entry `feedback_acm_gitops_pull_pattern.md` calls this out explicitly.

## ACM MultiClusterHub configuration

The hub runs ACM with a curated set of components enabled:

```yaml
apiVersion: operator.open-cluster-management.io/v1
kind: MultiClusterHub
metadata:
  name: multiclusterhub
  namespace: open-cluster-management
spec:
  disableHubSelfManagement: false
  overrides:
    components:
      - name: app-lifecycle
        enabled: true
      - name: cluster-lifecycle
        enabled: true
      - name: cluster-permission
        enabled: true
      - name: console
        enabled: true
      - name: grc
        enabled: true
      - name: multicluster-engine
        enabled: true
      - name: cluster-backup
        enabled: false
      - name: fine-grained-rbac
        enabled: false
      - name: insights
        enabled: false
      - name: multicluster-observability
        enabled: false
      - name: search
        enabled: false
      - name: siteconfig
        enabled: false
      - name: submariner-addon
        enabled: false
      - name: volsync
        enabled: false
```

What's enabled:

- `app-lifecycle` — Application/Subscription/Channel and the ManifestWork distribution pipe.
- `cluster-lifecycle` — `ManagedCluster`, `ManagedClusterSet`, Hive and Cluster API provisioning.
- `cluster-permission` — `ClusterPermission` CR for spoke RBAC distribution.
- `console` — ACM UI under the hub's OpenShift console.
- `grc` — Governance / Policy / Compliance pillar.
- `multicluster-engine` — registration, work, placement controllers.

What's disabled (will be re-enabled per tracked issue when added):

- `cluster-backup` — ACM backup/restore. Planned re-enable under [§4.4 ACM cluster backup](/docs/03-openshift-platform/04-acm-multicluster/04-acm-cluster-backup/).
- `multicluster-observability` — Thanos-backed federation. Not in scope; observability lives on the spoke.
- `search`, `insights`, `siteconfig`, `submariner-addon`, `volsync` — not in scope for this fleet.
- `fine-grained-rbac` — leveraging coarse-grained `ClusterRoleBinding` via `cluster-permission` instead.

## RHACS Central on the hub

RHACS Central runs on the hub because:

- Central is the policy server and UI for the whole fleet; running it on the management cluster keeps its blast radius low.
- The hub has LVMS for the Central, Central DB, and Scanner V4 PVCs. Moving Central to the spoke would have meant either making the spoke depend on the hub (wrong direction for the pull model) or using ODF for Central (defeats the storage-light hub posture for no operational gain).

RHACS Central reads pull-secret config from a hub Secret seeded by ESO from Vault (`secret/platform/rhacs/central-htpasswd`), and the init-bundle for the spoke `SecuredCluster` is generated by hitting Central's `/v1/cluster-init/init-bundles` API and pushing the resulting `kubectlBundle` Secrets into Vault for ESO delivery on the spoke (memory: `reference_rhacs_init_bundle_via_api.md`). The bundle is never committed.

## Hub Argo CD configuration

Single Argo CD instance per OpenShift GitOps default: `openshift-gitops/openshift-gitops`. It is bound to `cluster-admin` so it can install operators and write cluster-scoped resources on the hub itself. The hub Argo CD's `cluster-admin` posture is recorded in [ADR 0019](/docs/06-architecture-decisions/) and the memory entry `feedback_acm_gitops_pull_pattern.md`.

Hub Argo serves three roles:

1. Reconciles `clusters/hub-dc-v6/` from internal GitLab via the `hub-dc-v6-bootstrap` Application.
2. Hosts the `spoke-cluster-config-pull` ApplicationSet that generates per-spoke Applications.
3. Hosts the AppProject `hub-dc-v6-platform` that restricts hub Argo to the LAN GitLab source.

## Image supply on the hub

The hub uses the same Nexus mirror-registry as the spoke. IDMS / ITMS / CatalogSources for the hub live under:

```text
clusters/hub-dc-v6/platform/image-mirrors/
clusters/hub-dc-v6/platform/catalogs/
clusters/hub-dc-v6/platform/admission-policies/   # OperatorHub disable + future VAP
```

Default external OperatorHub sources are disabled (`OperatorHub/cluster.spec.disableAllDefaultSources: true`). The `cs-redhat-operator-index-v4-20` CatalogSource is the source for ACM, MCE, GitOps, LVMS, RHACS, ESO, and cert-manager Subscriptions.

> Note: per `feedback_acm_gitops_pull_pattern.md`, the hub's IDMS/ITMS were applied during the day-1 install bootstrap from oc-mirror output, and the durable manifests under `clusters/hub-dc-v6/platform/image-mirrors/` represent the same state. Reproducing the hub from scratch means re-running install bootstrap, then GitOps takes over.

## Health snapshot commands

```bash
K_HUB=/home/ze/.kube/configs/hub-dc-v6.kubeconfig

# Cluster baseline
oc --kubeconfig "$K_HUB" get nodes -o wide
oc --kubeconfig "$K_HUB" get clusterversion version
oc --kubeconfig "$K_HUB" get co | awk \
  'NR==1 || $3 != "True" || $4 != "False" || $5 != "False"'

# OpenShift GitOps
oc --kubeconfig "$K_HUB" -n openshift-gitops-operator get sub,installplan,csv,pods
oc --kubeconfig "$K_HUB" -n openshift-gitops get argocd,pods,route

# ACM / MCE
oc --kubeconfig "$K_HUB" -n open-cluster-management get sub,installplan,csv
oc --kubeconfig "$K_HUB" get multiclusterhub -A
oc --kubeconfig "$K_HUB" get managedcluster

# LVMS + RHACS
oc --kubeconfig "$K_HUB" -n openshift-storage get pods   # LVMS
oc --kubeconfig "$K_HUB" -n stackrox get pods            # RHACS Central + ScannerV4 + DB

# Bootstrap Application
oc --kubeconfig "$K_HUB" -n openshift-gitops get app hub-dc-v6-bootstrap \
  -o jsonpath='{.status.sync.status}{" "}{.status.health.status}{" "}{.status.sync.revision}{"\n"}'
```

Expected steady state:

- All three masters `Ready`.
- ClusterVersion `4.20.18`, `Available=True`, `Progressing=False`.
- No degraded ClusterOperators.
- All operator namespaces healthy.
- `MultiClusterHub` `Running`; `ManagedCluster/spoke-dc-v6` `Available=True`.
- `hub-dc-v6-bootstrap` Application `Synced Healthy`.
