This section documents compliance on `spoke-dc-v6` — the workload cluster — using the OpenShift Compliance Operator, the PCI-DSS profile, and the evidence chain that lands in MinIO. The hub (`hub-dc-v6`) doesn't run PCI scans; per ADR 0020 hub provides ACM/GitOps/governance evidence only, and PCI target is the workload cluster.

## What this is — and isn't

The Compliance Operator runs Red Hat-provided compliance profiles against an OpenShift cluster and produces machine-readable PASS/FAIL/MANUAL evidence. The PCI-DSS profile (v4.0) is one of those profiles. We use it to produce **OpenShift platform evidence**, not a formal Report on Compliance.

This is explicit per ADR 0020 (PCI-DSS baseline on spoke-dc-v6) and the compliance-implementor handbook:

| Inside scope | Outside scope |
| --- | --- |
| Reproducible OpenShift Compliance Operator profile evidence | A formal PCI DSS Report on Compliance |
| Documented cluster hardening | An Attestation of Compliance |
| Documented GitOps guardrails | A QSA sign-off |
| Sanitized evidence for an auditor / internal reviewer | Proof that applications are PCI compliant |
| Known residual risks + manual evidence requirements | Proof that the full Cardholder Data Environment is correctly scoped |

The output supports a PCI assessment; it does not substitute for one.

## Architecture

<Whiteboard client:load data={{ nodes, edges }} height={640} />

Reading the diagram:

1. **Compliance Operator** runs in `openshift-compliance`. It owns two ProfileBundles (`ocp4`, `rhcos4`) shipped with the operator image.
2. **ProfileBundles** materialize a set of `Profile` CRs. We use `ocp4-pci-dss-4-0` and `ocp4-pci-dss-node-4-0`.
3. **ScanSetting** `workers-storage` controls *how* scans run — worker-scheduled result server, RBD-backed raw result PVC, rotation policy.
4. **ScanSettingBinding** ties the profiles to the setting. Creating the binding kicks off a `ComplianceSuite`.
5. **ComplianceSuite** spawns one `ComplianceScan` per profile; each emits one `ComplianceCheckResult` per rule and one `ComplianceRemediation` per failed-but-fixable rule.
6. **Raw result PVC** is written by the result server (RWO, RBD, worker-scheduled). Sanitized exports land in MinIO `developer-ci-evidence/pci-dss/`.
7. **Remediations**, when applied, take effect via `MachineConfig` (node-level fixes like FIPS, SSHd config, sysctls) or direct cluster config (OAuth, Ingress, etcd encryption).

## The phase chain (PCI-0 → PCI-5)

Milestone #30 (`spoke-dc-v6 PCI-DSS Compliance Baseline`) breaks the work into a six-phase chain. Each phase has its own GitHub issue and must close before the next begins.

| Phase | Issue | Goal |
| --- | --- | --- |
| `PCI-0` | #108 | Port old `spoke-dc` PCI lessons into the current baseline; finalize ADR coverage (delivered ADR 0020). |
| `PCI-1` | #109 | Day-zero / day-one prereq validation. FIPS, vTPM, LUKS, ODF encryption, etcd encryption, OAuth, kubeadmin, IPv6 disabled-for-cluster-traffic. |
| `PCI-2` | #110 | Deploy Compliance Operator + PCI-DSS scan desired state through GitOps. Phase A done 2026-05-10 (operator + ProfileBundles VALID). |
| `PCI-3` | #111 | Hardening + tailoring (etcd encryption, OAuth inactivity, ODF route TLS tailoring, etc.). |
| `PCI-4` | #112 | Run suite, remediate failures, record final state. |
| `PCI-5` | #113 | Sanitize evidence, hand off. |

Don't skip phases. If a later phase looks easy, still prove prior gates are satisfied; leave the evidence on the prior issue.

## Why we ship this through GitOps

Per ADR 0025 (GitOps-only operations) and the compliance-implementor handbook, **every** PCI-relevant CR lives in `platform-gitops`:

| Resource | Location |
| --- | --- |
| `Namespace/openshift-compliance` | `clusters/spoke-dc-v6/compliance/namespace.yaml` |
| `OperatorGroup` | `clusters/spoke-dc-v6/compliance/operatorgroup.yaml` |
| `Subscription/compliance-operator` | `clusters/spoke-dc-v6/compliance/subscription.yaml` |
| `ScanSetting/workers-storage` | `clusters/spoke-dc-v6/compliance/scansetting-workers-storage.yaml` |
| `ScanSettingBinding/pci-dss` | `clusters/spoke-dc-v6/compliance/scansettingbinding-pci-dss.yaml` |
| `TailoredProfile/*` (if any) | `clusters/spoke-dc-v6/compliance/tailored-*.yaml` |

Direct `oc apply` is break-glass only. The Argo Application `spoke-dc-v6-cluster-config` owns the layer and reconciles within ~3 minutes.

## The non-negotiable prerequisites

Carried forward from old `spoke-dc` to ADR 0020:

| Prerequisite | What it means | Verification |
| --- | --- | --- |
| FIPS enabled from install | `fips: true` in install-config + butane `openshift.fips: true` | `oc get nodes -o jsonpath='{.items[*].status.nodeInfo.kernelVersion}'` includes `+fips` (or check `/proc/sys/crypto/fips_enabled`) |
| vTPM on VM masters | required for LUKS unlock and TPM2-bound clevis | hypervisor config |
| TPM2-backed LUKS on physical worker root | not a casual day-2 toggle | physical inspection / clevis status; recovery passphrase escrowed outside Git |
| ODF cluster-wide encryption | `storagecluster.spec.encryption.clusterWide=true` from cluster init | `oc -n openshift-storage get storagecluster -o yaml \| grep encryption -A3` |
| etcd at-rest encryption | `apiserver.spec.encryption.type: aesgcm` (OCP 4.20 prefers `aesgcm` over `aescbc`) | `oc get apiserver cluster -o yaml \| grep encryption` |
| OAuth inactivity timeout | both cluster-OAuth and per-OAuthClient | `oc get oauth cluster -o yaml`; `oc get oauthclient` |
| `kubeadmin` removed | after a real IdP wired | `oc get secret kubeadmin -n kube-system` returns `NotFound` |
| `ssh-rsa` only | NOT `ssh-ed25519` (FIPS-incompatible) | install-config inspection |
| IPv6 not used for cluster traffic | per ADR 0026 (4 verifiable invariants); host-kernel-disable forbidden (breaks OVN-K) | `oc get networks.config.openshift.io cluster -o yaml` |

If any of these is missing on `spoke-dc-v6`, scans will fail rules that have no in-cluster remediation. Phase PCI-1 (issue #109) is the prereq validation pass.

## Where the raw results go

Per ADR 0020 / the carried-forward `workers-storage` pattern:

```yaml
spec:
  nodeSelector:
    node-role.kubernetes.io/worker: ""
  roles:
    - master
    - worker
  rawResultStorage:
    size: 1Gi
    pvAccessModes: [ReadWriteOnce]
    rotation: 3
    storageClassName: ocs-storagecluster-ceph-rbd
```

The result-server pod is `nodeSelector`-pinned to worker nodes. The default (master placement) caused RBD attach failures on old `spoke-dc` because RBD volumes can't attach to nodes with the master taint; the worker-scheduled variant fixes it. `rotation: 3` keeps three scan generations in the PVC.

## Sanitized evidence to MinIO

Exports land in MinIO `developer-ci-evidence/pci-dss/` per ADR 0020. Each scan cycle produces:

- ComplianceSuite name, status, timestamp
- per-scan PASS/FAIL/MANUAL/NOT-APPLICABLE counts
- the platform-gitops Git SHA under which the scan ran
- the residual-risk list for FAIL and MANUAL findings
- (sanitized) raw rule output — secrets and identifying details redacted

The retention follows the `developer-ci-evidence` bucket lifecycle: see §9 (MinIO).

| Symptom | Cause | Subpage |
