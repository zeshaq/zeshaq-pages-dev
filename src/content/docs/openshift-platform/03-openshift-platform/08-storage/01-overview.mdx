The fleet runs two distinct storage stories — one on each cluster — and a third "object" plane that bridges both. This page is the map.

## The two clusters, two storage models

| Cluster | Role | Storage system | Why |
| --- | --- | --- | --- |
| `hub-dc-v6` | management plane (ACM, OpenShift GitOps, Compliance Operator) | **LVMS** (Logical Volume Manager Storage) | Compact 3-master VM topology with no physical disk pool; LVMS turns each master's local volume group into a `StorageClass` |
| `spoke-dc-v6` | workload cluster (apps, logging, tracing, registry, backup) | **ODF** (OpenShift Data Foundation; Ceph RBD + CephFS + NooBaa MCG) | Physical workers with dedicated data disks; full Ceph stack provides RBD block, CephFS shared filesystem, and S3 object |

This split is intentional and recorded in ADR 0004 (management-only hubs) and the post-install GitOps plan. The hub stays **storage-light**: it has just enough PVCs to hold ACM observability, GitOps repo cache, and operator state. The spoke does the heavy storage lifting.

## Reading the architecture

<Whiteboard client:load data={{ nodes: storageNodes, edges: storageEdges }} height={620} />

- **Hub side (left):** LVMS reads a local volume group on each control-plane VM and exposes it as a `lvms-vg1` StorageClass. PVCs on the hub bind to LVs sliced out of that VG. Cluster-local, no replication across nodes — adequate for hub workloads (kafka-less GitOps, no application data).
- **Spoke side (centre + right):** Local Storage Operator (LSO) discovers raw data disks on the physical worker nodes and creates `PersistentVolume` objects of type `localvolumeset`. ODF's `StorageCluster` consumes those PVs and stands up a Ceph cluster: OSDs on each worker's disks, MONs for quorum, MGRs for management. ODF then exposes three storage flavours through CSI:
  - **`ocs-storagecluster-ceph-rbd`** — block, `ReadWriteOnce`, the default StorageClass for general workloads.
  - **`ocs-storagecluster-cephfs`** — shared filesystem, `ReadWriteMany`, used where multiple pods need write access to the same volume (OADP backup repositories, some shared-cache patterns).
  - **NooBaa MCG / `s3.openshift-storage.svc`** — S3-compatible object endpoint, accessed by `ObjectBucketClaim` (OBC), used by Loki, Tempo, and Quay for blob storage.
- **Bottom row (clients):** the actual workloads that consume the storage. CNPG (CloudNativePG) takes RBD `ReadWriteOnce` for its Postgres data directories. Quay takes RBD for its operand state and an OBC for image-layer blobs. OADP takes a CephFS PVC plus a NooBaa OBC for backup state. Loki and Tempo take only OBC — no PVCs, S3-only.

## Why ODF on the spoke

The choice is constrained by the v6 design:

1. **Disconnected operation** — ODF ships as an OLM operator that we mirror via `oc mirror`; the same disconnected story already documented in [Section 6](/docs/openshift-platform/openshift-platform/disconnected-image-supply/architecture) applies.
2. **Hyperconverged density** — the three physical workers each carry data disks; ODF turns them into a unified storage layer without requiring a separate storage array.
3. **Three storage shapes in one operator** — RBD + CephFS + S3 from a single CSI install instead of three independent storage products. The single integration surface is operationally simpler.
4. **Compliance** — ADR 0020 requires cluster-wide ODF encryption from initialization (`storagecluster.spec.encryption.clusterWide: true`). Ceph's encrypted block-store gives this directly.
5. **Backup target** — ODF NooBaa is in-cluster S3, which OADP uses as its primary backup target during disconnected operation when external S3 isn't reachable.

The alternative, **Local Storage Operator alone** (no Ceph), was rejected for the spoke because:

- Local volumes are single-node; pod rescheduling away from the node carrying the data drops storage.
- No `ReadWriteMany` mode.
- No S3.

LSO is still in the picture, but as the **disk-discovery layer feeding ODF**, not as the storage layer itself.

## Why LVMS on the hub

LVMS is the right answer for compact OpenShift on VMs because:

- It doesn't require dedicated data disks per node — it reads a Linux volume group on each control-plane VM.
- The hub doesn't carry workload data; PVCs are small (Argo repo cache, GitOps state, observability ConfigMaps).
- It's a single CSI driver with no separate operator-state complexity.
- Each VM master gets its own LVMS LV — no inter-node replication, no quorum, no Ceph.

The trade-off: hub PVCs are local to one VM. If that VM is lost, the LV goes with it. For hub workloads this is acceptable; the hub's stateful workloads (ACM observability) are regenerated from GitOps state, and durable hub data lives off-cluster (the platform-gitops GitLab repo).

| Workload | Cluster | Storage type | StorageClass / OBC | Access mode | Notes |
| --- | --- | --- | --- | --- | --- |

| Open Liberty apps | spoke | block | `ocs-storagecluster-ceph-rbd` | RWO | per-pod state where required |
| OADP / Velero | spoke | shared FS + S3 | `ocs-storagecluster-cephfs` + NooBaa OBC | RWX + S3 | backup state + restic blobs |
| Loki (logs) | spoke | object | OBC `loki-storage` | n/a (S3) | bridged via ESO; see page 04 |

| Quay (image registry) | spoke | block + object | RBD PVC + OBC | RWO + S3 | image-layer blobs in S3 |
| Compliance Operator scan results | spoke | block | `ocs-storagecluster-ceph-rbd` | RWO | workers-storage ScanSetting (ADR 0020) |

## Default StorageClass posture

| Cluster | Default StorageClass | Rationale |
| --- | --- | --- |
| `hub-dc-v6` | `lvms-vg1` | the only StorageClass on the hub |
| `spoke-dc-v6` | `ocs-storagecluster-ceph-rbd` | block is the most common shape; if a workload needs CephFS it asks explicitly |

The default is set via the `storageclass.kubernetes.io/is-default-class: "true"` annotation. See [06-storage-class-defaults](/docs/openshift-platform/openshift-platform/storage/storage-class-defaults).

## Encryption baseline

ADR 0020 §Decision 8 + Carried-Forward Lessons makes two storage-encryption rules non-negotiable on `spoke-dc-v6`:

- **ODF cluster-wide encryption at cluster initialization** (`storagecluster.spec.encryption.clusterWide: true`).
- **TPM2-backed LUKS on physical worker root disks** with recovery passphrase custody outside Git.

The hub has its own requirements (vTPM on VM masters, FIPS at install time) but doesn't run ODF.

## Capacity baselines (illustrative)

The exact disk sizes are recorded in `opp-full-plat/plans/disconnected-rebuild/environments/dc-lab/odf-storage-inventory.md`. Per-cluster general ranges:

| Cluster | Storage | Approximate raw capacity |
| --- | --- | --- |
| `hub-dc-v6` | LVMS (sum across 3 masters) | 100s of GiB |
| `spoke-dc-v6` | ODF (sum across 3 workers, Ceph replication factor 3) | multiple TiB raw → ~1/3 usable |

Net usable depends on Ceph replication settings; ODF defaults to 3-way replication, so usable ≈ raw / 3.

## Boundaries and explicit non-goals

| Boundary | Why |
| --- | --- |
| No ODF on the hub | ADR 0004 — hub is management-only, storage-light |
| No LVMS on the spoke | LVMS doesn't give ReadWriteMany or S3; ODF already covers block |
| No external NFS server | adds an external dependency; ADR 0019 image rule extends to dependency surface — keep storage in-cluster |
| No public S3 (AWS / Wasabi / etc.) | disconnected operation — external S3 not assumed reachable; NooBaa is the S3 |
| No multi-cluster storage federation | each cluster owns its storage; cross-cluster data movement is via OADP backup-restore, not real-time replication |


- **ADR 0004** — management-only hubs.
- **ADR 0019** — Nexus-only image supply chain (covers ODF operand image references).
- **ADR 0020** — PCI baseline — ODF cluster-wide encryption, workers-storage ScanSetting.

## Failure modes (cross-page summary)

| Symptom | Root cause | See |
| --- | --- | --- |
| PVC `Pending` with no default class warning | StorageClass not annotated as default | page 06 |
| CSI driver pod `CrashLoopBackOff` | image-pull failure for an ODF operand digest | Section 6 — disconnected image supply |
| LokiStack `Warning Degraded` after fresh deploy | OBC Secret has wrong key shape | page 04 (OBC bridge) |
| Backup PVC `ReadWriteMany` requested but only RBD class available | wrong StorageClass | use CephFS for RWX workloads |
| Hub PVC won't reschedule to another VM | LVMS is node-local by design | accept the constraint or use a different cluster |
