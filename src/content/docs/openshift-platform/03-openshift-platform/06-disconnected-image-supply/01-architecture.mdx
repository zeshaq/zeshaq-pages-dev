This page is the map of the entire disconnected image supply chain for the dc-lab v6 fleet. The ADR that frames every rule on this page is **[ADR 0019: Nexus-Only Image Supply Chain](https://github.com/zeshaq/opp-full-plat/blob/main/adr/0019-nexus-only-image-supply-chain.md)** — every container image executed on `hub-dc-v6` or `spoke-dc-v6` must be available from the local Nexus mirror before a pod or operator is allowed to pull it.

## Reading the diagram

<Whiteboard client:load data={{ nodes: supplyNodes, edges: supplyEdges }} height={720} />

The flow has three logical stages and one strict invariant.

1. **Acquire** (top-left): a workstation VM named `oc-mirror` reaches the public registries (`registry.redhat.io`, `quay.io`, `icr.io`, `ghcr.io`, `nvcr.io`, `registry.connect.redhat.com`, `registry.access.redhat.com`) over the internet. This is the **only** host in the lab that ever touches the public image surface. Everything downstream pulls from the lab.
2. **Stage** (top-right): `oc mirror --v2` pushes every release image, operator bundle, operator index, and additional image into the Nexus VM at `mirror-registry.apps.sub.comptech-lab.com`, routed by the HAProxy edge and terminated against the lab's Let's Encrypt wildcard certificate.
3. **Deliver** (centre + bottom): `oc-mirror` emits a tarball of cluster resources — `ImageDigestMirrorSet`, `ImageTagMirrorSet`, `CatalogSource`, and (on 4.15+) `ClusterCatalog` objects. Those CRs land in the cluster through GitOps. Once present, `crio` on every node and `catalog-operator` in OLM rewrite every upstream image reference to the Nexus equivalent at pull time.

The invariant: **no production pod on a fleet cluster pulls from a public registry at runtime.** The IDMS-style rewrite makes the upstream name in a manifest cosmetic; the actual byte-stream comes from Nexus. This is what makes the fleet survive an offline weekend.

## Three Nexus endpoints, three roles

The single Nexus VM exposes three Docker Registry HTTP API v2 endpoints, each with a different purpose. They share the same TLS certificate (the lab wildcard) but are physically separate Nexus repositories with different cleanup policies and different role-based access.

| Public hostname | Purpose | Nexus repo | HAProxy backend | Who pushes | Who pulls |
| --- | --- | --- | --- | --- | --- |
| `mirror-registry.apps.sub.comptech-lab.com` | OpenShift platform mirror — release payload + operator index + operator bundles + additional images | `ocp-mirror` (hosted) | port `5000` | `oc-mirror` VM only | OpenShift nodes (crio) + OLM |
| `docker-group.apps.sub.comptech-lab.com` | Developer + CI base-image pulls (UBI, Open Liberty, Node.js, Docker Hub passthrough) | `docker-group` (group: `docker-dev-hosted` + `icr-proxy` + `redhat-proxy` + `dockerhub-proxy`) | port `5001` | nothing (group is read-only) | Jenkins / Tekton / developer workstations |
| `app-registry.apps.sub.comptech-lab.com` | CI-produced application image push and runtime pull (legacy Path A) | `docker-dev-hosted` (hosted) | port `5002` | Jenkins / Tekton | Argo-deployed workload pods |

This separation is non-negotiable per ADR 0019 §5 and the Nexus three-endpoint split (install / dev pull / app push). Crossing the streams (e.g., pushing an app image to `mirror-registry`) is a class of incident, not a tactical workaround: the OpenShift Image Set tooling owns the layout under `mirror-registry/openshift/...` and assumes it is the only author.

## What "image supply" includes

| Class | Examples | Lives where |
| --- | --- | --- |
| OpenShift release payload | `openshift/release-images@sha256:...`, machine-os-content, MCO daemonset images | `mirror-registry/openshift/release-images` |
| Operator catalog index | `cs-redhat-operator-index-v4-20`, `cs-certified-operator-index-v4-20` | `mirror-registry/...catalog/redhat-operator-index:v4.20` |
| Operator bundles + operands | `advanced-cluster-management`, `odf-operator`, every CSV-referenced image | `mirror-registry/redhat/...`, `mirror-registry/odf4/...` |
| Additional images | UBI9 base, Node.js 22 minimal, Open Liberty 25.0.0.6 ICR base | `mirror-registry/ubi9/...`, `mirror-registry/appcafe/open-liberty` |
| Developer base images | UBI9 (passthrough), upstream Node.js (passthrough), Dockerhub passthrough | `docker-group.apps...` (proxy repos) |
| Built application images | `team-platform/sample`, `smoke/readiness-probe` | `app-registry.apps...` |

The first four classes are platform-owned and live on `mirror-registry`. The last two are application-owned and live on `docker-group` and `app-registry`. The boundary is stable: app teams cannot push to `mirror-registry`, and platform code does not push to `app-registry`.

## Why this design, not the alternatives

| Alternative | Why we did not pick it |
| --- | --- |
| **In-cluster registry as day-zero mirror** | Circular dependency: the cluster cannot install without an image source, but the in-cluster registry only exists after install. ADR 0019 §3 forbids this for the day-zero mirror. The in-cluster `image-registry.openshift-image-registry.svc:5000` is allowed only for BuildConfig output. |
| **Mirroring release+operators directly to each cluster's local registry** | N clusters × N petabytes of duplication. The cluster registries are storage-light by design (LVMS on hub, ODF on spoke is for workloads). |
| **Direct internet pulls at runtime** | The lab WAN is slow and unreliable; an upstream outage stalls operator installs and image-stream tags. This is also explicitly disallowed by ADR 0019 — runtime registries must be in the approved allowlist. |
| **Quay Operator as the platform mirror** | Quay is an in-cluster operand. Same circular dependency as the in-cluster registry. We do run Quay later for app images (DEV-OCP-2.5 / #186), but it is a downstream choice, not the day-zero mirror. |
| **Skopeo + manual `docker push` instead of oc-mirror** | Skopeo is fine for one image. oc-mirror v2 handles the OpenShift release catalogue + N operator bundles + index regeneration + IDMS/ITMS emission in one idempotent pipeline. The tool exists to solve this exact problem. |

## Failure-mode digest

The session log under `opp-full-plat/SESSION_LOG.md` and the issue tracker have these recurring symptoms; the design above is largely a reaction to them.

| Symptom | Root cause | Fix | Prevention |
| --- | --- | --- | --- |
| ODF CSI pods enter `ImagePullBackOff` with `registry.redhat.io/odf4/...` error | ODF operand image not in the mirror, even though the operator bundle was | Mirror the exact CSV-listed operand digest; regenerate IDMS | Pin operand sub-packages (`mcg-operator`, `ocs-operator`, `cephcsi-operator`, etc.) explicitly in `ImageSetConfiguration` |
| Pod uses `ghcr.io/cloudnative-pg/...` and pull fails on spoke | IDMS rewrite missing for ghcr.io; allowlist passes but no Nexus copy exists | Add the digest to next mirror run; add ghcr.io to IDMS source list | Add cnpg-style operators to the mirrored set before installing Subscription |
| Pull from `mirror-registry` returns 401 from a workload pod | Pod's namespace has no pull secret linked to its ServiceAccount | `oc secrets link <sa> nexus-docker-pull --for=pull` | ESO + Vault bridging the pull-secret per namespace; see Section 4 — Secrets / ESO |
| `oc mirror --v2` aborts with `manifest unknown` mid-run | Upstream digest disappeared (CVE rebuild, garbage collection) | Refresh `oc-mirror` cache, re-run; consider bumping pinned version | Pin `minVersion == maxVersion` in ImageSetConfiguration so a re-run gives byte-identical output until the file changes |
| RHACS deploy-time policy blocks otherwise mirrored image | Allowlist entry missing on RHACS Central side (the host-only granularity layer) | Edit `DESIRED_REGISTRIES` and rerun `rhacs-update-img-allowlist.sh` | Two-layer allowlist always edited together — see page 05 |
