The security plane stacks four independent control layers — RHACS, VAP, NetworkPolicy, Pod Security — over the same workloads. Each catches a different class of risk, and they're designed so the failure of one still leaves a meaningful check standing. This page is the topology; subpages drill into each component.

## What's in the security plane

| Layer | Tool | What it gates | Failure leaves still standing |
| --- | --- | --- | --- |
| Build-time image vulnerability | Trivy → DefectDojo (out of scope here; see §6 supply-chain docs) | known CVEs at build | RHACS deploy-time policy will still flag if a known-bad lands |
| Deploy-time policy (CR-level) | RHACS Central deploy policies | latest tag, missing limits, privileged, CAP_SYS_ADMIN, required labels | VAP allowed-image-registries still denies disallowed registries |
| Admission-time registry allowlist | `ValidatingAdmissionPolicy/allowed-image-registries` | any image not from an allow-listed prefix | RHACS deploy-time policy alerts; runtime collector observes |
| Admission-time RHACS gate | RHACS admission controller (per cluster) | RHACS-policy-violating Pods | Kubernetes-native VAP still denies registry violations |
| Pod Security Standards | OpenShift built-in (per-namespace) | SCC-equivalents (`restricted`, `restricted-v2`, etc.) | RHACS Privileged Container policy alerts |
| Runtime detection | RHACS Collector (DaemonSet) | unexpected processes, cryptomining, lateral movement | none — last line of defense; relies on alerting |
| Network egress/ingress | per-tenant default-deny NetworkPolicy set | east-west traffic, lab-perimeter egress | RHACS network-flow detection still surfaces anomalies |

The doctrine: **don't rely on a single layer**. If a Trivy fail-gate is bypassed via a build-side exception, RHACS will still alert at deploy. If RHACS Sensor goes degraded, the VAP still denies non-allow-listed registries at the apiserver. Defense in depth is non-negotiable on a PCI-target cluster.

## Architecture

Reading the diagram:

- **Central + Scanner V4 + Postgres** all run on `hub-dc-v6` in `stackrox` ns. Central is the policy engine and UI; Scanner V4 is the image-scanning service (replaced the older `Scanner` + `Scanner-DB`); Postgres holds policies, alerts, and history.
- **Sensor + Admission controller** run on each cluster (hub manages itself in addition to the spoke). The Sensor is the long-lived pull-mode agent that maintains an mTLS connection to Central; the Admission controller webhook is registered with the local kube-apiserver and consults the Sensor for policy evaluation.
- **Collector** runs as a DaemonSet on the spoke. It's the kernel-mode (or eBPF) probe that produces runtime events (process, network, file) and feeds them to the Sensor.
- **VAP** is Kubernetes-native admission — independent of RHACS. The cluster-wide allowed-image-registries policy lives in `platform-gitops` per cluster.
- **NetworkPolicy** defaults live in every tenant overlay (the W1.C template, DEV-OCP-4.2 / #197). Default-deny + named allows.
- **Vault** stores the per-Sensor init-bundle (per `reference_rhacs_init_bundle_via_api.md`); ESO materializes the three TLS Secrets (`collector-tls`, `sensor-tls`, `admission-control-tls`) into `stackrox` on each cluster.

## RHACS at a glance

RHACS is structured as five distinct pods/Deployments that talk over a hub-spoke pull model identical in spirit to RHACM's klusterlet:

1. **Central** — the brain. Stores policies, alerts, images. UI lives here.
2. **Sensor** (per cluster) — the on-cluster controller. Long-lived mTLS connection back to Central. Reads Kubernetes resources, evaluates policy locally, sends events upstream.
3. **Admission controller** (per cluster) — registers a webhook with the kube-apiserver. Evaluates Pods on `CREATE` / `UPDATE` against deploy-time policies. Talks to Sensor for current policy snapshot.
4. **Collector** (DaemonSet) — host-level agent producing runtime events (eBPF or kernel-mode driver).
5. **Scanner V4** — pulls images from registries (using the secured cluster's pull credentials), runs Clair-derived static analysis, attaches vulnerability/CVE data to image objects in Central.

The hub-spoke split makes RHACS work across air-gapped, NAT-ed, and disconnected configurations — the Sensor initiates the TLS connection; Central never reaches *into* a managed cluster.

## Where RHACS *isn't* the enforcement layer

A few categories are explicitly **not** RHACS-enforced even though Central can:

- **Image vulnerability gating** — Trivy + DefectDojo on the build path. RHACS sees the CVE landscape but the build gate is what fails a deploy of a known-bad image. Doing both creates double-blocking that slows incident triage. RHACS still *alerts* on deploy-time CVE policy if one slips past.
- **NetworkPolicy enforcement** — RHACS observes network flows and can suggest NetworkPolicies, but the actual enforcement is the per-tenant NetworkPolicy set from DEV-OCP-4.2.
- **Allowed-registry admission** — RHACS *also* runs a deploy-time `IMG-SUPPLY-3` policy, but the **primary** registry gate is the Kubernetes-native VAP. The RHACS policy is the corroborating layer (lower granularity on the registry path) and the source of audit-friendly alerts in the Central UI.
