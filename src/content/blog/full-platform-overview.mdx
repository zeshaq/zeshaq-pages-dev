This is the deployed platform shape so far. The control plane is intentionally
split: GitHub tracks project governance and public documentation, while the
OpenShift runtime desired state lives in internal GitLab and is reconciled by
OpenShift GitOps.

## Current operating model

- `hub-dc-v6` is the management cluster.
- `spoke-dc-v6` is the active workload cluster.
- Internal GitLab is the source of truth for OpenShift desired state.
- GitOps is the normal operating path; `oc` and console access are break-glass.
- `oc-mirror` is reserved for OpenShift release and operator image supply.
- Nexus is reserved for application and CI image/package caching.
- Vault is the external secrets authority, consumed by External Secrets
  Operator.

## Deployed so far

| Layer | Current state |
| --- | --- |
| Management | `hub-dc-v6`, OpenShift GitOps, ACM, MCE |
| Security | RHACS Central on hub, SecuredCluster on hub and spoke |
| Secrets | Vault-backed External Secrets stores on both clusters |
| Storage | LVMS on hub; LSO and ODF on spoke |
| Backup | OADP baseline on spoke |
| Compliance | Compliance Operator and PCI scans on spoke, with remediation ongoing |
| Certificate management | cert-manager on both clusters |
| Image supply | oc-mirror v2 for OpenShift content; Nexus for non-OCP image/package supply |

## Next expansion

The next platform work is to finish RHOAI mirror recovery and then install
remaining operators through GitOps only, including OpenShift Virtualization,
Pipelines, observability, service mesh, Quay, and future application runtime
operators.
